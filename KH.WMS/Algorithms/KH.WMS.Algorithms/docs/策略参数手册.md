# 策略参数手册（KH.WMS.Algorithms）

> 每个策略的**必填 🔴 / 可选 🟡** 参数、默认值、对结果的影响、产出与链位置。
> 现状基于 A2 之后（出库 = 单步 InventoryAllocation；整托/分区偏好下沉为库存分配的 StepParams）。
> 配套：[算法包表结构契约.md](./算法包表结构契约.md)

## 读法

- **来源**：`ContextData`=调用方 `context.SetData(key,value)` 注入；`属性`=`IPolicyContext` 强类型属性；`StepParams`=策略链步骤 JSON（`cfg_strategy_chain_step.StepParams`）；`服务`=DI 构造注入。
- **缺失行为**：库存类（FIFO/FEFO/BATCH/UTIL）必填缺失 → 抛异常 → `Failure`（#40）；货位/上架/下架类多为静默 `Skipped`。
- **通用出库偏好**（仅 InventoryAllocation 族）：写在 StepParams JSON，由基类 `ApplyPreferences` 解析，做稳定排序（不改变同组内 FIFO/FEFO 顺序）。
  - `EnableWholePalletFirst`（bool，默认 false）：能一次满足整单的库位排前。
  - `ZonePriorityList`（`[{ZoneCode,Priority}]`，默认无）：配置的库区按 Priority 升序排前，未列入排后。
- **结果读取**：单步/非流水线链读 `policyResult.Output as XxxResult`；流水线链读 `context.GetData<XxxResult>(XxxOutput.RESULT)`。

## 一、库存分配策略（InventoryAllocation）

链位置：出库链 = 单步 `InventoryAllocation`（默认走 `CHAIN_INV_DEFAULT`=FIFO）。消费 `WarehouseCode/MaterialCode/RequiredQty`；产出 `InventoryAllocationResult`（写 `InventoryAllocationOutput.RESULT`）。

### FIFO — 先进先出
全仓 `GetByFIFOAsync`（按入库时间，NULL 排后），叠加偏好后遍历扣减至满额。

| 参数 | 来源 | 必填 | 默认 / 缺失 | 影响 |
|---|---|---|---|---|
| IInventoryQueryService | 服务 | 🔴 | 缺失→Failure | 库存查询 |
| WarehouseCode | ContextData(回退属性) | 🔴 | 空→Failure | 限定仓库 |
| MaterialCode | ContextData | 🔴 | 空→Failure | 物料 |
| RequiredQty | ContextData | 🔴 | 缺失=0→Skipped | 需求量；<=0 跳过 |
| EnableWholePalletFirst | StepParams | 🟡 | false | 整托优先排序 |
| ZonePriorityList | StepParams | 🟡 | 无 | 分区优先排序 |

产出：`InventoryAllocationResult{Items, TotalAllocatedQty, RequiredQty, IsFullySatisfied, ShortageQty}`。

### FEFO — 先过期先出
全仓 `GetByFEFOAsync`（按过期日，NULL 排后），其余同 FIFO。参数表同 **FIFO**（必填 4 + 可选 偏好 2）。

### BATCH — 按批次出库
先取指定批次（`GetByBatchAsync`），不足按 FIFO 补充（排除该批次）。

| 参数 | 来源 | 必填 | 默认 / 缺失 | 影响 |
|---|---|---|---|---|
| IInventoryQueryService / WarehouseCode / MaterialCode / RequiredQty | — | 🔴 | 同 FIFO | — |
| TargetBatchNo | ContextData | 🟡 | 无 | 非空：优先该批次，不足 FIFO 补；空：退化为纯 FIFO（含无批次库存） |
| EnableWholePalletFirst / ZonePriorityList | StepParams | 🟡 | — | 同 FIFO |

### UTILIZATION_PRIORITY — 利用率优先
`GetByFIFOAsync` 取候选，按「可用量升序 → 入库时间升序」排序（优先清空低库存库位），再叠偏好。

参数表同 **FIFO**（必填 4 + 可选 偏好 2）。Reason 标 `"利用率优先"`。

> 通用产出：`Items[].Reason` 标 `"先进先出"/"先过期先出"/"指定批次(...)"/"FIFO补充"/"利用率优先"`；每项含 `InventoryDetailId/HeaderId/LocationId/LocationCode/ContainerCode/MaterialId/BatchNo/SerialNo/AvailableQty/AllocatedQty/ExpiryDate/ManufactureDate/InboundTime/Priority`。

## 二、货位分配策略（LocationAllocation）

链位置：上架流程末端；流水线模式（`PipelineMode=true`），每步从上游候选筛选。统一经基类 `PostProcess`：`SortRules` 排序 → `EnableDoubleDeep` 双深过滤 → `MaxRecommendCount` 截断。产出 `LocationAllocationResult{Locations}`（写 `LocationAllocationOutput.RESULT`）。

### 通用可选（StepParams JSON，4 个 LocationAllocation 策略共享）
| 参数 | 默认 | 影响 |
|---|---|---|
| `MaxRecommendCount` | 20 | 末尾 `Take(N)` 截断 |
| `SortRules` `[{Field,Direction}]` | 无(不排序) | 多字段稳定排序；Field: AisleNo/RowNo/ColNo/LayerNo/Depth/Score/ZoneCode/LocationCode |
| `EnableDoubleDeep` | false | true 启用前后排约束过滤 |
| `DoubleDeepMode` | FRONT_FIRST | FRONT_FIRST/BACK_FIRST/PAIR（需 EnableDoubleDeep=true） |

### ABC_CLASS — ABC 分类货位
按物料 ABC（A/B/C）匹配库区 + 货位级 ABC 校验。

| 参数 | 来源 | 必填 | 默认 / 缺失 | 影响 |
|---|---|---|---|---|
| WarehouseId | 属性(回退ContextData) | 🔴 | <=0→Skipped | 仓库范围 |
| IWarehouseQueryService | 服务 | 🔴 | null→不工作 | 查 ABC 分类/库区 |
| ILocationQueryService | 服务 | 🔴 | null→空结果 | 查空闲货位/双深 |
| MaterialId | 属性 | 🟡 | 无→默认"C" | 查物料 ABC；缺失按 C 类分配 |
| MaterialAbcClass | ContextData | 🟡 | 无 | 强制覆盖 ABC 分类（优先级最高） |
| 通用 StepParams 4 项 | StepParams | 🟡 | 见上 | 排序/双深/截断 |

### CATEGORY_ZONE — 品类分区货位
按物料品类匹配专用库区。

| 参数 | 来源 | 必填 | 默认 / 缺失 | 影响 |
|---|---|---|---|---|
| WarehouseId | 属性 | 🔴 | <=0→Skipped | 仓库范围 |
| IWarehouseQueryService + ILocationQueryService | 服务 | 🔴 | null→Skipped | 查库区/货位 |
| MaterialCategoryId | 属性 | 🟡 | 无 | 无：全仓找空货位；有：按映射/`CATEGORY_{id}`匹配库区 |
| CategoryZoneMapping `{分类ID:ZoneCode}` | StepParams | 🟡 | 空 | 显式 categoryId→ZoneCode 映射；未配按 ZoneType 约定 |
| 通用 StepParams 4 项 | StepParams | 🟡 | 见上 | 排序/双深/截断 |

### CENTRALIZED — 集中存储货位
同物料/批次优先放到已有库存货位的**同巷道**附近。

| 参数 | 来源 | 必填 | 默认 / 缺失 | 影响 |
|---|---|---|---|---|
| ILocationQueryService | 服务 | 🔴 | null→Skipped | 查货位/附近 |
| WarehouseId | 属性 | 🔴 | <=0→Skipped | 仓库 |
| MaterialId | 属性 | 🔴 | <=0→Skipped | 决定"已有同物料库存货位" |
| TargetBatchNo | ContextData | 🟡 | 无 | 叠加批次维度（同批次聚集更紧） |
| MaxNearbyCount | ContextData | 🟡 | 10 | 链首查库：每个已有货位附近回查上限 |
| 通用 StepParams 4 项 | StepParams | 🟡 | 见上 | 排序/双深/截断 |

> 流水线模式：有上游候选则按同巷道加分（+50），不丢弃候选；无上游且无已有库存 → `FallbackToAnyEmpty`。

### DOUBLE_DEEP — 双深货位
为双深货架（Depth=1前排/2后排）按模式分配，防堵仓。**自带双深逻辑**（`GetMode` 只读 `DoubleDeepMode`，不读基类 `EnableDoubleDeep`）。

| 参数 | 来源 | 必填 | 默认 / 缺失 | 影响 |
|---|---|---|---|---|
| IWarehouseQueryService + ILocationQueryService | 服务 | 🔴 | null→Skipped | 查库/双深判断 |
| WarehouseId | 属性 | 🔴 | <=0→Skipped | 仓库 |
| DoubleDeepMode | StepParams | 🟡 | FRONT_FIRST | FRONT_FIRST(前排优先,前排空则丢弃后排)/BACK_FIRST/PAIR |
| SortRules | StepParams | 🟡 | 内置(LayerNo→Depth→RowNo→ColNo) | 自定义排序 |
| MaxRecommendCount | StepParams | 🟡 | 20 | 截断 |
| ZoneId | 属性 | 🟡 | 无 | 链首：有值按库区查(含占用,双深判断准)；空→全仓空货位 |
| 上游候选 | ContextData | 🟡 | 无(链首) | 有：流水线精筛；无：查库 |

> 通用产出：`LocationRecommendation{LocationId,LocationCode,AisleNo,RowNo,ColNo,LayerNo,Depth,Side,ZoneCode,Score,Reason}`；Score≈`max(100-层*2-(Depth-1)*5,0)`（DOUBLE_DEEP 另按模式 ±10/深度）。

## 三、上架策略（Putaway）

### DEFAULT_PUTAWAY — 默认上架
四步：入库口 → 目标库区(ABC) → 最闲巷道 → 路径。不读 StepParams JSON（读固定 ContextData 键）。

| 参数 | 来源 | 必填 | 默认 / 缺失 | 影响 |
|---|---|---|---|---|
| WarehouseId | 属性(回退ContextData) | 🔴 | <=0→空结果 | 仓库；缺则各步跳过 |
| IWarehouseQueryService | 服务 | 🔴 | null→跳过查询 | 入库口/库区/巷道 |
| InboundStationId / InboundStationCode | ContextData | 🟡 | 无 | 显式入库口，优先于 DocTypeId 匹配（#5） |
| DocTypeId | ContextData | 🟡 | 无 | 未传入库口时按单据类型匹配（#5 回退） |
| EnableZonePartition | ContextData | 🟡 | true | false→跳过库区匹配，全库选巷道 |
| EnableAbcClass | ContextData | 🟡 | true | false→跳过 ABC，回退首个存储库区 |
| MaterialId | 属性 | 🟡 | 无 | 查 ABC 分类（需 ZonePartition+AbcClass 开启） |
| PathOptimization | ContextData | 🟡 | S_SHAPE | S_SHAPE/Z_SHAPE/U_SHAPE |

产出：`PutawayResult{InboundStationId/Code, TargetZoneId/Code, AisleId/Code, Route, PathOptimization, RequireLocationAllocation=true}`（写 `PutawayOutput.RESULT`）。链需后续接货位分配策略。

## 四、下架策略（Picking）

### DEFAULT_PICKING — 默认下架
把上游库存分配结果转下架任务，决策出库口，生成取货/送达路径（`CranePathOptimizer`）。不读 StepParams JSON。

| 参数 | 来源 | 必填 | 默认 / 缺失 | 影响 |
|---|---|---|---|---|
| InventoryAllocationResult（上游） | ContextData | 🔴 | 无→空任务 | 库存分配结果，转 PickingTask |
| IWarehouseQueryService | 服务 | 🔴 | null→不匹配出口 | 出口匹配 |
| DestinationStationId / Code | ContextData | 🟡 | 无 | 显式出口，跳过自动匹配 |
| DestinationZoneId / Code | ContextData | 🟡 | 无 | 直传结果；自动匹配时由端口 ZoneId 回填 |
| DocTypeId | ContextData | 🟡 | 无 | 自动匹配首选：`GetPortByDocTypeAsync(OUTBOUND)` |
| SourceAisleNo | ContextData | 🟡 | 0 | DocTypePort 未命中时：`GetOutboundPortByAisleAsync`（巷道→接驳口→输送线→站台） |
| PathOptimization | ContextData | 🟡 | S_SHAPE | S_SHAPE/Z_SHAPE/U_SHAPE，决定任务排序与 PickRoute |

产出：`PickingResult{Tasks[], DestinationStationId/Code, DestinationZoneId/Code, PickRoute, DeliveryRoute, PathOptimization, RequireInventoryAllocation=true}`（写 `PickingOutput.RESULT`）。每任务 `IsFullPallet = AllocatedQty>=AvailableQty`（按实际取走量，#38）。

## 附：调用方注入速查

| 流程 | 调用方 | 执行入口 | 必须注入的服务 |
|---|---|---|---|
| 上架 | `InboundContainerBindService` | `ExecuteAsync(Putaway)` | SVC_LOCATION_QUERY + SVC_WAREHOUSE_QUERY |
| 货位分配(Phase2) | `TaskHeaderService` | `ExecuteAsync(LocationAllocation)` | SVC_LOCATION_QUERY + SVC_WAREHOUSE_QUERY |
| 出库(选库存) | `OutboundAllocationService` | `ExecuteAsync(InventoryAllocation)` | SVC_INVENTORY_QUERY + SVC_WAREHOUSE_QUERY |
| 下架 | `OutboundAllocationService` | `ExecuteAsync(Picking)` | SVC_WAREHOUSE_QUERY |

> 出库偏好配置示例（写到库存分配链步骤的 `StepParams`）：
> ```json
> {"EnableWholePalletFirst": true, "ZonePriorityList": [{"ZoneCode":"PICKING","Priority":1},{"ZoneCode":"STORAGE","Priority":2}]}
> ```
