## 2. 内容原则与"代码 + 3-5 行解读"标准

### 2.1 一段代码 + 3-5 行解读的统一模板

```js
// src/utils/request.js（真实代码片段）
const instance = axios.create({ baseURL: '/api', timeout: 30000 })
```

**解读**（4 行标准）：
- **为什么 baseURL 是 `/api` 而非 `http://localhost:9291`**：开发环境走 Vite 代理，生产环境由 Nginx/IIS 反代；硬编码后端地址会让前后端地址耦合在不同环境反复改。
- **为什么用 `axios.create` 而非全局 `axios.defaults`**：`create` 返回独立实例，多实例场景下不会污染其他第三方库；KH.WMS 全局只用一个实例。
- **timeout 30s 的取舍**：KH.WMS 报表查询、Excel 导出偶尔会超 10s；30s 是经验值，不至于让用户长时间无响应。
- **踩坑**：误把 `baseURL` 写成 `/api/`（带尾斜杠）会导致 `/api/auth/login` 变成 `/api//auth/login`，部分后端会 404。

## 4. A/B/P 线在主文档中的统一使用规范

### 4.1 定义（§0 总谱统一出现）
- **A 线（架构纵线）**：跟踪一个 HTTP 请求从浏览器进入到渲染返回的完整链路，关心 axios、路由守卫、Pinia、Layout 等横切基础设施。
- **B 线（业务纵线）**：在某个业务域（如 basedata）下，从后端就绪到新增一个完整可运行页面的全过程，关心 API 封装、CRUD 范式、字典、权限。
- **P 线（PDA 纵线）**：手持端特有场景，关心紧凑布局、扫码、实时反馈、错误重试。

### 4.2 每节引用规范
- 每节顶部写「本节以 X 线为例」一句话，X ∈ {A, B, P}
- 正文中出现"以 X 线为例"时不需要重复说明 X 的含义（§0 总谱已定义）
- A/B/P 三个标签可同时出现（如"§3.1 A 线 LoginView + B 线 brand 骨架"）
