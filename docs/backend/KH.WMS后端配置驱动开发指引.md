## 2. 配置能力总览

| 能力 | 主要模块 | 典型实体/接口 | 使用场景 |
| --- | --- | --- | --- |
| 全局配置 | `ConfigModule` | `CfgGlobalConfig` | 系统开关、流程参数、业务阈值 |
| 系统参数 | `SystemModule` | `SysParameter` | 登录、默认值、系统级行为 |
| 字典 | `SystemModule` | `SysDictType` / `SysDictItem` | 状态展示、枚举翻译、下拉选项 |
| 扩展字段 | `ConfigModule` | `CfgExtField` / `CfgExtFieldType` | 动态表单、业务实体补充字段 |
| 单据类型 | `ConfigModule` | `CfgDocumentType` | 区分采购入库、销售出库、调拨等流程 |
| 单据状态 | `ConfigModule` | `CfgDocumentStatus` | 单据状态展示与流程约束 |
| 单据流程 | `ConfigModule` | `CfgDocumentTypeProcess` | 定义单据流转节点 |
| 状态规则 | `ConfigModule` | `CfgDocumentTypeRule` | 控制状态流转是否合法 |
| 编码规则 | `ConfigModule` | `CfgCodeRule` / `CfgCodeSequence` | 单据号、库位编码、业务编号 |
| 站台/库区映射 | `ConfigModule` | `CfgDocTypePort` | 按单据类型选择站台、口位、库区 |

## 3. 什么时候优先配置化

优先配置化：

- 会因客户、仓库、单据类型变化而变化的规则。
- 需要运营或实施人员维护的选项。
- 页面字段、下拉选项、展示状态。
- 单据状态流转、编码规则、站台/库区映射。
- 未来可能按仓库、物料、货主、客户切换的业务规则。

可以写代码：

- 稳定的技术流程，例如统一响应、事务、日志、认证。
- 不应被用户配置改变的系统约束。
- 性能敏感且规则稳定的基础查询逻辑。

## 4. SystemModule 与 ConfigModule 边界

`SystemModule` 偏系统通用能力：

- 用户、角色、权限、菜单。
- 字典、系统参数。
- 操作日志、附件、文件。

`ConfigModule` 偏业务配置能力：

- 扩展字段。
- 单据类型、状态、流程、规则。
- 编码规则。
- 仓储类型、站台类型、库位类型等业务配置。

判断口诀：

- “系统管理通用能力”放 `SystemModule`。
- “WMS 业务可变规则”放 `ConfigModule`。
- “策略计算规则”放 `KH.WMS.Algorithms`。

## 5. 扩展字段与动态表单

扩展字段常用于不修改表结构也要扩展页面字段的场景。

后端入口：

- 配置实体：`KH.WMS/Entities/KH.WMS.Entities/Config/CfgExtField.cs`
- 配置服务：`KH.WMS/Modules/ConfigModule/KH.WMS.Modules.ConfigModule/Services`
- Contract：`KH.WMS/Modules/ConfigModule/KH.WMS.Modules.ConfigModule/Contracts/CfgExtFieldContract.cs`

前端联动入口：

- 表单配置接口通常是模块 Controller 的 `/form-config`。
- 前端通过 `src/utils/useExtFields.js` 或页面表单配置消费扩展字段。

开发约定：

- 页面需要动态字段时，优先查是否已有扩展字段能力。
- 不要为了新增少量客户定制字段立刻改实体和数据库表。
- 后端返回给前端的表单配置应保持稳定结构，避免每个模块自定义一套格式。

扩展字段落地步骤：

1. 确认字段属于哪个业务表和哪个区域，例如物料主表 HEADER。
2. 在扩展字段配置中维护字段编码、名称、类型、是否必填、排序。
3. 后端业务 Controller 提供或复用 `/form-config`。
4. 前端表单读取配置，合并固定字段和扩展字段。
5. 保存时将扩展字段值按项目约定写入扩展数据结构。

培训时要强调：扩展字段不是替代所有表结构变更。如果字段参与高频查询、索引、核心业务计算，应评估是否成为正式实体字段。

## 6. 字典与系统参数

字典适合：

- 页面下拉选项。
- 状态颜色与文案。
- 后端枚举展示映射。

系统参数适合：

- 系统默认值。
- 开关类配置。
- 与业务模块无强绑定的全局参数。

注意：

- 字典不应承载复杂业务规则。
- 业务专属配置优先使用 ConfigModule。
- 字典变更后前端需要刷新缓存，前端缓存入口在 `src/stores/dict.js`。

## 7. 单据类型、状态与流程

单据相关配置用于控制不同业务流程：

- `CfgDocumentType` 定义单据类型。
- `CfgDocumentStatus` 定义状态。
- `CfgDocumentTypeProcess` 定义流程节点。
- `CfgDocumentTypeRule` 定义状态流转规则。

开发约定：

- 新增单据类型时，同时检查状态、流程、编码规则、站台映射。
- Service 中不要硬编码“从 A 状态直接跳 B 状态”的规则。
- 状态流转失败应返回明确业务错误，方便前端展示和排查。

状态流转培训示例：

```text
入库单创建
  -> 待收货
  -> 已收货
  -> 上架中
  -> 已完成
```

如果客户要求“质检不通过必须进入异常处理”，不要在 `InboundOrderService` 中直接写死跳转，应先确认：

- 是否已有单据状态可复用。
- 是否需要新增单据流程节点。
- 是否需要新增状态规则。
- 前端是否需要展示新状态和操作按钮。

## 8. 编码规则

编码规则常用于：

- 入库单号。
- 出库单号。
- 任务号。
- 库位编码。
- 批次或业务流水号。

开发约定：

- 需要连续编号、前缀、日期、流水号时，优先查 `CfgCodeRule` 和 `CfgCodeSequence`。
- 不要在业务 Service 中手写 `DateTime.Now + Random` 生成关键业务编号。
- 编码生成失败时应记录业务上下文，便于按单据排查。

## 9. 配置读取与服务调用建议

配置读取要遵守三个原则：

1. 业务 Service 只关心“配置结果”，不关心配置表如何组织。
2. 可缓存的配置应通过配置服务统一缓存和刷新，避免每次请求打数据库。
3. 配置缺失必须有明确策略：默认值、阻断错误或降级逻辑。

推荐结构：

```text
Controller
  -> BusinessService
  -> ConfigResolverContract / ConfigService
  -> ConfigModule entities
```

不推荐：

```text
BusinessService
  -> 直接 Queryable<CfgXxx>
  -> 到处拼配置逻辑
```

配置缺失处理建议：

| 配置类型 | 缺失处理 |
| --- | --- |
| 展示类字典 | 可返回空列表或默认文案 |
| 编码规则 | 应阻断并提示配置缺失 |
| 状态流转规则 | 应阻断并提示流程配置缺失 |
| 可选业务开关 | 可使用默认值 |
| 安全类配置 | 应阻断启动或请求 |

## 10. 培训案例：新增可配置质检开关

需求：不同入库单据类型可配置是否需要质检。

推荐落地：

1. 在 ConfigModule 中确认是否已有单据类型配置或全局配置可承载。
2. 如果是按单据类型变化，优先挂在单据类型配置或流程规则上。
3. InboundModule 在收货完成后读取配置结果。
4. 如果需要质检，状态进入“待质检”；否则进入“待上架”。
5. 前端根据状态展示“质检确认”或“分配库位/上架”按钮。
6. 验收时分别验证需要质检和不需要质检两种单据类型。

这个案例能训练成员区分：

- 规则是否配置化。
- 配置由哪个模块维护。
- 业务模块如何读取配置。
- 前端如何按配置结果展示流程。

## 11. 配置评审清单

开发前评审：

- 这条规则是否会因客户、仓库、单据类型变化？
- 是否已有配置表能承载？
- 是否需要前端页面维护？
- 是否需要缓存？
- 缺失时应该默认、报错还是降级？
- 是否会影响状态流转或编码生成？

代码评审：

- 是否把可变规则写死在 Service？
- 是否绕过 ConfigModule 直接查配置表？
- 是否有清晰错误提示？
- 是否考虑前端字典/扩展字段刷新？
- 是否有测试或手工验收场景？

## 12. 常见错误

| 错误 | 后果 | 正确做法 |
| --- | --- | --- |
| 把字典当业务规则表 | 后续规则扩展困难 | 复杂规则放 ConfigModule 或 Algorithms |
| 状态流转硬编码 | 改流程必须发版 | 走单据流程/规则配置 |
| 客户定制字段直接加表字段 | 数据模型膨胀 | 优先评估扩展字段 |
| 页面下拉写死 | 字典修改无法同步 | 走后端字典和前端 dict Store |
| 编码规则写在 Service | 多模块编号不一致 | 走编码规则和序列 |

## 13. 课后检查

- 能判断一个新需求应改配置还是改代码。
- 能说明 `SystemModule` 与 `ConfigModule` 的边界。
- 能说出扩展字段、字典、单据状态、编码规则的典型用途。
- 能指出硬编码状态流转和硬编码编号的风险。
