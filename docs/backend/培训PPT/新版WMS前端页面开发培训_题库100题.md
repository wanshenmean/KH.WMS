### 简答题参考要点

91. 后端 `[Route("api/material")]` → `module="material"` → 菜单 `component="basedata/material"` → `src/views/basedata/material.vue` → `permissionPrefix="bd:material"`。它们分别决定 API 路径、动态路由加载和按钮权限；任一项不一致都会导致 404、页面找不到或按钮权限混乱。（依据：§2、§3.1、§7）

92. 建页面文件；先写 KhPage 骨架并验证菜单打开；按查询 DTO 写 `searchModel/searchColumns`；按列表 DTO 写 `tableColumns`；按新增/编辑 DTO 写 `formColumns`；配置 `crudOperations` 与 `permissionPrefix`；按需加入动作按钮和扩展字段；最后验证查询、新增、编辑、删除、导出、权限。（依据：§3.4）

93. KhTable 传入 `pageNum/pageSize/业务字段/sortConditions`；`useCrudApi.pageList` 通过 `buildPageQuery` 将 `pageNum` 转成 `pageIndex`，保留排序，并把业务字段按列类型转换为 `{ field, operator, value }` 的 `filters`。表头筛选与查询区条件合并为同一 filters 数组。（依据：§4.2）

94. KhPage 只透传普通表格属性和插槽，当前不会继续 emit KhTable 事件，并会过滤 `onXxx`。状态交互推荐用 `actionButtons`；必须放开关等内联交互时，用 `type: 'slot'` 自己渲染并处理。（依据：§4.5、§14）

95. 四类配置都以同一 `prop` 为核心：它决定查询参数、`row[prop]`、`formData[prop]`、校验字段和插槽名称。拼写或 DTO 对不齐会导致查询条件丢失、表格无值、表单无法回填或提交字段错误。（依据：§5.1、§5.2、§5.3、§5.4）

96. 页面字段随业务配置变化、需要动态加载时使用 `useExtFields`。`load` 用于把列表/详情的扩展数据回显为扁平字段；`beforeSubmit` 用于从表单提取、清理扩展字段并提交到 `extDataRaw`。（依据：§8.2、§8.3、§8.4）

97. 主从表请求体通常是 `{ order, lines }`，编辑又需完整详情，内置单表 CRUD 难以满足。应由 KhPage 管主列表，关闭内置 create/update，手写 KhDialog；弹窗中用 KhForm 管主表、KhEditableTable 管明细，提交时处理明细归一化、主从组装、扩展字段和 loading，成功后提示并 reload。（依据：§9.1—§9.6）

98. 内置表单模式传 `form-columns/form-model`，KhDialog 自动创建并校验 KhForm，校验通过将 data 传给 `confirm`，适合简单单表。自定义内容模式适合主从表和复杂布局，KhDialog 不知道插槽内容，confirm 不带表单数据也不自动校验，需手动 validate 和取数。（依据：§10.4）

99. 父组件：用 `v-model` 控制显隐，传当前行必要 props，监听 `success` 刷新 KhPage。子组件：用 props/emit 和可写 computed 实现 v-model，自己负责请求、校验、loading，成功后关闭并 `emit('success')`。（依据：§10.2、§10.3）

100. 示例：页面目录、菜单 component、module、permissionPrefix、searchModel 初始值、各类 prop、字典编码与 tag 写法、自定义按钮 permission、成功后 reload、事件边界、主从表手写流程、扩展字段 load/beforeSubmit、loading、手动 validate、Object.assign、错误提示。示例解释：页面放进 `components` 会被动态路由过滤；`permissionPrefix` 错会导致按钮不显示或权限混乱。（依据：§13、§14）
