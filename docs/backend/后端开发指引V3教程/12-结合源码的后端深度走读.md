# 第 12 章 结合源码的后端深度走读

> 本章不是新增一套开发规则，而是把前面章节讲过的启动、注册、CRUD、事务、AOP、Contract 和业务流程，放回当前后端源码里走一遍。读完以后，应该能从一个接口请求一路追到 Controller、Service、仓储、事务、跨模块 Contract、策略引擎和最终数据库写入。

## 12.1 先建立三条源码主线

KH.WMS 后端代码可以先按三条主线理解。

| 主线 | 入口 | 关键源码 | 读源码时要回答的问题 |
| --- | --- | --- | --- |
| Web 启动主线 | `KH.WMS.Server/Program.cs` | `KH.WMS.Core/Setup/*`、`KH.WMS.Core/Middlewares/*` | 请求进来之前，哪些基础设施、过滤器、中间件、Controller 扫描规则已经生效 |
| 通用 CRUD 主线 | `CrudController<TEntity>` | `CrudService<TEntity>`、`IRepository<T,TKey>`、`IUnitOfWork`、`DetailSaveService` | 普通表维护为什么只写很少代码，底层怎样完成查询、写入、事务和主从表保存 |
| 业务流程主线 | 各模块 `Services/*Service.cs` | `Contracts/*Contract.cs`、`KH.WMS.Contracts/*`、策略引擎、状态校验器 | 入库、出库、任务、库存这种多模块流程，事务边界、状态流转和并发防护在哪里 |

这三条线互相叠加。比如一次出库分配请求：

```text
HTTP 请求
  -> Program.cs / UseCustomMiddleware 建好的管道
  -> OutboundAllocationController
  -> OutboundAllocationService.AllocateAsync
  -> PolicyEngine 执行库存分配策略
  -> IInventoryContract.LockInventoryAsync 锁定库存
  -> IOutboundContract 更新出库单状态
  -> UnitOfWork 提交事务
  -> ApiResponse / TraceId 返回前端
```

所以排查问题时不要只盯 Controller。这个项目真正的业务含义通常在 Service、Contract、状态校验器和策略链里。

## 12.2 启动链路: `Program.cs` 做了什么

启动入口是 `KH.WMS/KH.WMS.Server/Program.cs`。它不是普通模板生成的简单 `Program.cs`，里面有几个对业务开发很关键的动作。

### 12.2.1 宿主与容器

`Program.cs` 首先创建 `WebApplicationBuilder`，然后调用：

```csharp
builder.Host.UseWindowsService(...)
builder.Host.AddSerilog()
builder.Host.UseServiceProviderFactory(new AutofacServiceProviderFactory())
builder.RegisterModule(new ServiceExtensions())
```

这里有两个重点。

1. 项目运行时既可以作为普通 Web 进程，也可以作为 Windows Service。
2. DI 容器不是只用默认 `Microsoft.Extensions.DependencyInjection`，而是接入 Autofac。后面 `[RegisteredService]`、接口代理、AOP 拦截器，都依赖 Autofac 和 Castle DynamicProxy。

真正扫描服务的入口在：

```text
KH.WMS.Core/DependencyInjection/ServiceExtensions.cs
KH.WMS.Core/DependencyInjection/ServiceRegistrar.cs
```

`ServiceExtensions.Load` 会通过 `AssemblyService.GetReferencedAssemblies()` 拿到当前解决方案引用到的程序集，再交给 `ServiceRegistrar` 扫描带特性的类。

### 12.2.2 基础设施注册

`builder.Services.AddInfrastructure(builder.Configuration, builder.Environment)` 是 Core 层的统一基础设施入口，对应：

```text
KH.WMS.Core/Setup/ServiceCollectionSetup.cs
```

当前它主要注册：

| 能力 | 源码入口 | 对业务开发的影响 |
| --- | --- | --- |
| SqlSugar | `AddSqlSugarSetup` | 仓储、UnitOfWork、事务、实体建表都依赖它 |
| 缓存 | `AddCacheSetup` | Config、权限、字典、限流等会用缓存 |
| JWT 认证 | `AddAuthenticationSetup` | 请求进入业务前会先完成 Token 校验 |
| 日志 | `AddLoggingSetup`、`AddSerilog` | 异常、TraceId、AOP 调用链都写日志 |
| MiniProfiler | `AddMonitoringSetup` | SQL 和方法调用耗时可观察 |
| Swagger | `AddApiDocumentationSetup` | Controller 被发现后才能在文档中出现 |
| CORS | `AddCustomCors` | 前后端分离调试时影响跨域 |
| 限流 | `AddRateLimiting` | 服务已注册，当前中间件默认未启用 |
| 全局 MVC 过滤器 | `ApiAuthorizeFilter`、`TraceIdResultFilter` | 认证和响应 TraceId 在全局生效 |

注意：`Program.cs` 里还单独调用了 `AddControllers`，并添加 `GlobalExceptionFilter` 和 `TraceIdResultFilter`。所以 TraceId 相关过滤器在两个入口都能看到，读代码时要知道它不是业务 Controller 自己加的。

### 12.2.3 Controller 扫描

业务模块 Controller 能出现在 Swagger，不是因为它们在 Server 项目下，而是因为 `Program.cs` 里显式配置了 `ApplicationPartManager`：

```csharp
var moduleAssemblies = AssemblyService.GetReferencedAssemblies()
    .Where(a => a.GetName().Name?.Contains(".Modules.") == true || a.GetName().Name == "KH.WMS.Config");

foreach (var assembly in moduleAssemblies)
{
    apm.ApplicationParts.Add(new AssemblyPart(assembly));
}
```

这解释了一个常见问题：新增 Controller 以后 Swagger 看不到，不一定是路由写错，也可能是程序集没有被 Server 引用，或者 Controller 所在程序集没有进入 ApplicationPart。

### 12.2.4 请求管道

请求管道入口在：

```text
KH.WMS.Core/Setup/MiddlewareSetup.cs
```

`UseCustomMiddleware` 的顺序大致是：

```text
异常处理
  -> License 校验
  -> HTTPS 重定向
  -> 静态文件
  -> CORS
  -> 请求日志
  -> 路由
  -> MiniProfiler
  -> Authentication
  -> Authorization
  -> MapControllers / MapRazorPages
```

业务开发要记住两点。

1. Controller 之前已经经过异常、License、CORS、日志、认证授权这些基础设施。
2. 如果一个请求没有进入 Controller，要先确认中间件层有没有拦截，比如 License、鉴权、CORS、路由。

`Program.cs` 里还在管道前段启用了请求体缓冲：

```csharp
context.Request.EnableBuffering();
```

这对 `ExtDataCrudController<TEntity>` 很重要，因为它会在模型绑定后重新读取原始 JSON，从里面提取 `extDataRaw`。

## 12.3 自动注册与 AOP: `[RegisteredService]` 的真实含义

业务服务常见写法：

```csharp
[RegisteredService(ServiceType = typeof(IOutboundAllocationService))]
public class OutboundAllocationService(...) : CrudService<OutboundAllocationHeader>(...), IOutboundAllocationService
```

这个特性不只是“让 DI 能注入”。它在当前项目里还有两个含义。

1. 注册到 Autofac 容器。
2. 默认启用接口代理拦截器。

源码在 `ServiceRegistrar`：

```csharp
builder.RegisterType(implementationType)
    .As(serviceType)
    .InstancePerLifetimeScope()
    .EnableInterfaceInterceptors()
    .InterceptedBy(interceptors);
```

当前默认拦截器数组是：

```text
LoggingInterceptor
CachingInterceptor
ConfigValidationInterceptor
ExceptionInterceptor
PerformanceInterceptor
```

因此，当你通过接口调用服务方法时，可能会自动获得日志、缓存、配置校验、异常处理、性能监控能力。

### 12.3.1 为什么文档一直强调“通过接口调用”

Autofac 的 `EnableInterfaceInterceptors()` 作用在接口代理上。也就是说，AOP 生效的典型路径是：

```text
Controller 注入 IInboundContainerBindService
  -> Autofac 返回代理对象
  -> 调用 ContainerBindAsync
  -> ConfigValidationInterceptor / LoggingInterceptor 等拦截
  -> 真实 InboundContainerBindService.ContainerBindAsync 执行
```

如果在类内部直接调用自己的方法，或者直接 `new` 一个 Service，就不会经过接口代理。

这也是为什么跨模块调用要注入 `IInventoryContract`、`ITaskContract`、`IOutboundContract`，而不是直接引用对方模块的 Service 实现。

### 12.3.2 `ServiceType` 最好显式写

`ServiceRegistrar` 在没有指定 `ServiceType` 时，会取实现类的第一个接口：

```csharp
var interfaces = item.Type.GetInterfaces();
Type serviceType = interfaces.FirstOrDefault() ?? item.Type;
```

这在类只实现一个接口时问题不大。但业务服务经常同时继承基类、实现模块接口、泛型接口，靠“第一个接口”会增加不确定性。

推荐始终写成：

```csharp
[RegisteredService(ServiceType = typeof(IMyService))]
public class MyService(...) : CrudService<MyEntity>(...), IMyService
```

Contract 实现也一样：

```csharp
[RegisteredService(Lifetime = ServiceLifetime.Scoped, ServiceType = typeof(IInventoryContract))]
public class InventoryContract(IUnitOfWork unitOfWork) : IInventoryContract
```

### 12.3.3 什么时候用 `WithoutInterceptor`

`UnitOfWork` 的注册是：

```csharp
[RegisteredService(Lifetime = ServiceLifetime.Scoped, WithoutInterceptor = true)]
public class UnitOfWork : IUnitOfWork
```

这类底层基础设施不应被业务 AOP 拦截。否则日志、异常、缓存、性能拦截器内部又可能依赖 UnitOfWork 或日志服务，容易产生循环依赖或递归调用。

适合 `WithoutInterceptor = true` 的对象通常是：

| 类型 | 原因 |
| --- | --- |
| UnitOfWork、DbContext、Logger 等底座服务 | 拦截器自身可能依赖它们 |
| 纯查询拦截器状态的服务 | 避免自己查询自己 |
| 高频、无业务语义的内部工具 | 拦截收益低，反而增加噪音 |

普通业务 Service、Contract、Validator 不建议随意关闭拦截器。

## 12.4 一个标准 CRUD 请求的底层路径

普通 CRUD Controller 继承：

```text
KH.WMS.Core/Controllers/CrudController.cs
```

服务基类是：

```text
KH.WMS.Core/Services/CrudService.cs
```

典型子类示例：

```text
KH.WMS.Modules.WarehouseModule/Services/LocationService.cs
KH.WMS.Config/Services/CfgDocumentFieldService.cs
KH.WMS.Modules.SystemModule/Services/SysUserService.cs
```

### 12.4.1 Controller 只做 HTTP 入口

`CrudController<TEntity>` 固定提供这些端点：

| 端点 | 方法 | 调用的 Service 方法 |
| --- | --- | --- |
| `GET {id}` | `GetById` | `GetByIdAsync` |
| `POST pagelist` | `GetPagedList` | `GetPagedListAsync` |
| `GET all` | `GetAll` | `GetListAsync` |
| `POST create` | `Create` | `CreateAsync` |
| `POST update` | `Update` | `UpdateAsync` |
| `DELETE delete/{id}` | `Delete` | `DeleteAsync` |
| `DELETE batch` | `BatchDelete` | `BatchDeleteAsync` |
| `PUT status/{id}` | `SetStatus` | `SetStatusAsync` |
| `POST export` | `Export` | `ExportAsync` |
| `POST import` | `Import` | `ImportAsync` |
| `GET template` | `DownloadTemplate` | `DownloadTemplateAsync` |

所以子 Controller 通常只需要写路由、注入接口、继承基类。

```csharp
[Route("api/location")]
public class LocationController(ILocationService service)
    : CrudController<MdLocation>(service)
{
}
```

如果 Controller 里出现大量业务逻辑，通常说明职责放错了。Controller 应该只处理 HTTP 语义，业务规则要下沉到 Service。

### 12.4.2 查询链路

`CrudService.GetPagedListAsync` 的核心流程是：

```text
BuildQueryExpression(query)
  -> ExpressionHelper.BuildFilter(query.Filters)
  -> 两个表达式 And 合并
  -> _repository.AsQueryable().Where(...)
  -> ApplyAdditionalQuery(queryable, query)
  -> ApplySorting(queryable, query.SortConditions)
  -> ToPageListAsync(pageIndex, pageSize, total)
  -> AfterQueryAsync(query, items)
  -> ApiResponse.Ok(new { items, total })
```

这几个钩子的含义不同：

| 钩子 | 适合放什么 |
| --- | --- |
| `BuildQueryExpression` | 模块默认过滤条件，比如租户、业务状态、只查当前仓库 |
| `ApplyAdditionalQuery` | 联表、Include/Mapper、额外动态查询 |
| `ApplySorting` / `ApplyDefaultSorting` | 默认排序或特殊排序 |
| `AfterQueryAsync` | 查询后的字段补充、字典翻译、轻量派生数据 |

不要把所有逻辑都塞进 `AfterQueryAsync`。能在数据库过滤的条件尽量放在表达式或 queryable 上，否则分页和总数容易失真。

### 12.4.3 新增链路

`CrudService.CreateAsync` 的核心流程是：

```text
BeginTransactionAsync
  -> BeforeCreateAsync
  -> FillAuditFields
  -> _repository.AddAsync(entity)
  -> _detailSaveService.SaveDetailsAsync(entity, true)
  -> _detailSaveService.SaveOneToOneAsync(entity, true)
  -> AfterCreateAsync
  -> CommitAsync
```

异常时会 `RollbackAsync` 后继续抛出异常。

开发时优先重写这些钩子，而不是复制整个 `CreateAsync`：

| 场景 | 推荐钩子 |
| --- | --- |
| 新增前校验唯一性 | `BeforeCreateAsync` |
| 新增前补业务字段 | `BeforeCreateAsync` |
| 新增后清缓存 | `AfterCreateAsync` |
| 新增后写其他派生表 | `AfterCreateAsync`，但要确保还在同一事务语义内 |

### 12.4.4 更新链路

`CrudService.UpdateAsync` 不是直接把前端实体整条覆盖到数据库，而是：

```text
BeginTransactionAsync
  -> GetEntityOrThrowAsync(entity.Id)
  -> BeforeUpdateAsync(entity)
  -> CopyProperties(entity, existing)
  -> CopyDetailProperties(entity, existing)
  -> FillAuditFields(existing, false)
  -> _repository.UpdateAsync(existing)
  -> DetailSaveService 保存从表和一对一
  -> AfterUpdateAsync(existing)
  -> CommitAsync
```

这里最容易忽略的是 `CopyProperties` 的策略：它跳过主键、跳过级联导航属性，并且只复制非 null 值。

这意味着：

| 前端传值 | 结果 |
| --- | --- |
| 传非 null | 会覆盖数据库字段 |
| 不传或传 null | 默认不覆盖原字段 |
| 想把字段清空为 null | 需要单独设计清空语义，不能直接依赖默认 Update |

如果某个业务明确需要支持“置空字段”，要在子 Service 中重写更新逻辑，或者增加专用 DTO 和专用方法。

### 12.4.5 删除与批量删除

单条删除会先判断实体是否存在级联导航属性：

```text
HasCascadeNavigationProperties()
  -> 有级联导航: GetByIdWithNavAsync + DeleteWithNavAsync
  -> 无级联导航: GetByIdAsync + DeleteAsync
```

批量删除走：

```text
BeforeBatchDeleteAsync(ids)
  -> _repository.DeleteAsync(ids)
  -> AfterBatchDeleteAsync(ids)
```

如果删除要做“引用检查”，优先放在 `BeforeDeleteAsync` 或 `BeforeBatchDeleteAsync`。

例如“物料已经有库存，不能删除”，这种判断不应该放 Controller。

## 12.5 事务边界: CRUD 内置事务与流程事务

当前项目有两种事务入口。

| 事务入口 | 源码 | 适用场景 |
| --- | --- | --- |
| CRUD 基类内部手动事务 | `CrudService.CreateAsync/UpdateAsync/DeleteAsync/BatchDeleteAsync` | 普通表维护 |
| `[Transaction]` Action Filter | `TransactionAttribute`、`TransactionActionFilter` | Controller 或 Action 级别的流程接口 |

### 12.5.1 UnitOfWork 支持事务状态

`IUnitOfWork` 暴露：

```csharp
bool HasActiveTransaction { get; }
int TransactionDepth { get; }
IDbContext DbContext { get; }
IRepository<T, TKey> GetRepository<T, TKey>()
```

这说明 Service 可以在同一个 UnitOfWork 下拿多个仓储，完成跨表写入。

业务流程里常见写法：

```csharp
await _unitOfWork.BeginTransactionAsync();
try
{
    // 多个仓储和多个 Contract 操作
    await _unitOfWork.CommitAsync();
}
catch
{
    await _unitOfWork.RollbackAsync();
    throw;
}
```

### 12.5.2 `[Transaction]` 会避开已有事务

`TransactionActionFilter` 里有一段关键逻辑：

```csharp
if (_unitOfWork.HasActiveTransaction)
{
    await next();
    return;
}
```

也就是说，如果 Service 已经开始事务，Action Filter 不会再重复包一层。这样可以避免普通 CRUD 和流程接口之间出现混乱的嵌套事务。

### 12.5.3 流程型 Service 要特别注意早返回

不少业务流程方法会在事务内做校验，然后返回 `ServiceResult.Fail(...)`。如果已经 `BeginTransactionAsync()`，早返回前必须确认是否需要 `RollbackAsync()`。

例如下面这种结构是安全的：

```csharp
await _unitOfWork.BeginTransactionAsync();
try
{
    if (!result.Success)
    {
        await _unitOfWork.RollbackAsync();
        return result;
    }

    await _unitOfWork.CommitAsync();
    return ServiceResult.Ok();
}
catch
{
    await _unitOfWork.RollbackAsync();
    throw;
}
```

如果在事务已开启后直接 `return ServiceResult.Fail(...)`，事务释放行为就取决于底层 DbContext/连接生命周期，读起来也不清晰。新增流程接口时应把“失败返回前是否回滚”作为检查项。

## 12.6 ExtData: 动态字段不是普通 JSON 字段

动态扩展字段相关代码主要有两套。

| 场景 | 代码 |
| --- | --- |
| 普通带 `ExtData` 的 CRUD | `ExtDataCrudController<TEntity>` |
| 单据头和明细动态字段 | `CfgDocumentFieldExtContract`、`CfgExtFieldContract`、入库/出库订单 Controller |

### 12.6.1 `ExtDataCrudController` 为什么要重读 Body

`ExtDataCrudController<TEntity>` 在 `Create/Update` 中会：

```text
HttpContext.Request.Body.Position = 0
  -> 读取原始 JSON
  -> 找 extDataRaw
  -> 写入实体 ExtData 属性
  -> 调用 service.CreateAsync / UpdateAsync
```

它依赖 `Program.cs` 里的 `EnableBuffering`。如果删掉请求体缓冲，模型绑定后 Body 流不能稳定重读，`extDataRaw` 就可能取不到。

### 12.6.2 `GetById` 会把 ExtData 展平

`ExtDataCrudController.GetById` 会把实体序列化成 `JsonObject`，再把 `ExtData` JSON 展开合并进去。

这让前端编辑页可以像读普通字段一样读动态字段。

但要注意：

1. 如果实体没有 `ExtData` 属性，继承 `ExtDataCrudController` 没意义。
2. 如果 `ExtData` JSON 格式异常，当前实现会忽略展开，不会抛错。
3. `GetPagedList` 的展开当前主要由前端或导出逻辑处理，不要误以为所有列表接口都会自动展开。

### 12.6.3 单据类 ExtData 更推荐走 Config Contract

入库单、出库单的创建 DTO 里有：

```text
ExtDataRaw
LineExtDataRaw
ExtDataFlattened
LineExtDataFlattened
```

对应 Controller 会通过 `ICfgDocumentFieldExtContract` 查询字段配置，再把前端提交的动态字段序列化到实体 `ExtData`。

这比直接把整个 JSON 原样塞进数据库更可控，因为它能结合单据类型、字段层级、是否可处理等配置。

## 12.7 Contract: 跨模块协作的门面

跨模块接口定义放在：

```text
KH.WMS/Contracts/KH.WMS.Contracts
```

实现放在各自业务模块里：

```text
BaseDataModule/Contracts/MaterialContract.cs
WarehouseModule/Contracts/LocationContract.cs
InventoryModule/Contracts/InventoryContract.cs
TaskModule/Contracts/TaskContract.cs
InboundModule/Contracts/InboundOrderContract.cs
OutboundModule/Contracts/OutboundContract.cs
```

这种设计有三个目的。

1. 调用方只依赖稳定接口，不直接引用对方模块的 Service。
2. 被调用模块自己维护内部实体和状态细节。
3. 多模块流程可以共享同一个 `IUnitOfWork`，形成一个业务事务。

### 12.7.1 Contract 应该暴露能力，不暴露内部表维护

好的 Contract 方法通常是业务动作：

```text
IInventoryContract.LockInventoryAsync
IInventoryContract.GenerateInventoryFromPutawayAsync
IInventoryContract.MoveInventoryLocationAsync
ITaskContract.CreatePutawayTaskAsync
IOutboundContract.UpdateOrderStatusAsync
IInboundOrderContract.MarkBindAsPuttingAwayAsync
```

不推荐把对方模块的 CRUD 方法全部搬到 Contract 里。Contract 是跨模块门面，不是远程 Repository。

### 12.7.2 Contract 里的仓储来自同一个 UnitOfWork

例如 `InventoryContract` 构造函数只注入 `IUnitOfWork`：

```csharp
public class InventoryContract(IUnitOfWork unitOfWork) : IInventoryContract
```

方法内部通过：

```csharp
var detailRepo = unitOfWork.GetRepository<InvInventoryDetail, long>();
```

拿仓储。

当 `OutboundAllocationService.AllocateAsync` 先开始事务，再调用 `IInventoryContract.LockInventoryAsync`，库存锁定就和出库分配明细、订单状态更新处于同一个事务语义里。

这就是“流程 Service 控制事务，Contract 提供能力”的核心。

## 12.8 入库链路源码走读

入库主链路可以按这几个动作看：

```text
创建入库单
  -> 组盘 ContainerBindAsync
  -> 请求上架 RequestPutAwayAsync / RequestPutAwayByWcsAsync
  -> 创建上架任务 CreatePutawayTaskAsync
  -> WCS 分配实际货位 AllocatePutawayLocationAsync
  -> WCS 回传完成 CompleteTaskByWcsAsync
  -> 生成库存 GenerateInventoryFromPutawayAsync
```

### 12.8.1 组盘: `InboundContainerBindService.ContainerBindAsync`

源码：

```text
KH.WMS.Modules.InboundModule/Services/InboundContainerBindService.cs
```

核心步骤：

1. 开启事务。
2. 收集容器号和入库明细行 ID。
3. 调用 `IContainerContract.RegisterContainersAsync` 自动注册容器。
4. 对容器记录加 `UpdLock`，序列化同一容器的并发组盘。
5. 校验容器没有活跃组盘、没有库存、没有活跃任务。
6. 校验订单明细存在。
7. 按容器分组创建组盘头和组盘明细。
8. 更新容器状态为 `IN_USE`。
9. 检查入库单是否全部组盘，必要时推进订单状态。
10. 提交事务。

这里有两个深层设计点。

第一，并发防护不是靠前端禁按钮，而是靠数据库更新锁和事务内复核。第二，状态推进不是简单赋值，而是通过 `IDocumentStatusValidatorContract` 校验允许的流转。

### 12.8.2 请求上架: `RequestPutAwayAsync`

请求上架会从组盘头创建上架任务。

核心动作：

```text
查询组盘头和明细
  -> 校验 BindStatus == BOUND
  -> 执行上架策略 PolicyEngine.ExecuteAsync(PolicyType.Putaway)
  -> 根据策略结果找接驳口
  -> 调用 ITaskContract.CreatePutawayTaskAsync
  -> 调用 IInboundOrderContract.MarkBindAsPuttingAwayAsync
  -> 更新容器状态 MOVING
```

注意，上架任务创建时不一定已经有最终货位。代码里注释把它分成两个阶段：

| 阶段 | 动作 |
| --- | --- |
| Phase 1 | 上架策略确定库区、巷道、入库口或接驳口，创建任务 |
| Phase 2 | WCS 到接驳口后调用货位分配接口，分配实际货位 |

这就是为什么 `TaskHeaderService.AllocatePutawayLocationAsync` 独立存在。

### 12.8.3 上架完成: `TaskHeaderService.CompleteTaskByWcsAsync`

WCS 完成任务进入 `TaskHeaderService`：

```text
KH.WMS.Modules.TaskModule/Services/TaskHeaderService.cs
```

关键步骤：

1. 根据 `TaskNo` 或 `WcsTaskNo` 找任务。
2. 对任务行加 `UpdLock`，处理重复回调和并发回调。
3. 异常分支：任务标记失败，写 `TaskConfirm`，返回失败。
4. 正常分支：任务标记完成，写 `TaskConfirm`。
5. 根据任务类型执行后续逻辑。

对上架任务，后续逻辑包括：

```text
GenerateInventoryViaContractAsync
  -> 读取 TaskLine
  -> 通过 IInboundOrderContract 获取组盘明细和来源单号
  -> 构建 InventoryGenerationRequest
  -> IInventoryContract.GenerateInventoryFromPutawayAsync
  -> 回写 TaskLine.InventoryHeaderId
```

然后更新库位、容器、组盘状态。

这条链路说明：库存不是 WCS 回调直接写库存表，而是由任务完成后通过库存 Contract 统一生成。

## 12.9 出库链路源码走读

出库主链路可以按这几个动作看：

```text
创建出库单
  -> AllocateAsync 分配库存
  -> GenerateTasksAsync 生成拣货任务
  -> WCS 回传完成 CompleteTaskByWcsAsync
  -> MoveInventoryLocationAsync 移动库存位置
  -> 后续确认或扣减库存
```

### 12.9.1 出库分配: `OutboundAllocationService.AllocateAsync`

源码：

```text
KH.WMS.Modules.OutboundModule/Services/OutboundAllocationService.cs
```

核心步骤：

1. 校验出库单存在、状态允许、仓库存在、明细存在。
2. 开启事务。
3. 对出库单加 `UpdLock`，防止同一订单并发分配两次。
4. 遍历出库行。
5. 执行库存分配策略 `PolicyType.InventoryAllocation`。
6. 校验分配数量满足需求。
7. 调用 `IInventoryContract.LockInventoryAsync` 原子锁定库存。
8. 创建出库分配头和明细。
9. 调用 `IOutboundContract` 更新出库单和出库单行状态。
10. 提交事务。

这里最重要的是第 7 步。

`InventoryContract.LockInventoryAsync` 不是先查可用量再普通更新，而是用条件更新：

```text
UPDATE InvInventoryDetail
SET LockedQty = LockedQty + lockQty
WHERE Id = inventoryDetailId
  AND Qty - LockedQty >= lockQty
```

受影响行数为 0 就说明可用量不足或被并发占用。这样可以避免两个并发分配同时看到“可用库存足够”，最后超锁。

### 12.9.2 生成拣货任务: `GenerateTasksAsync`

出库分配后生成任务的逻辑：

```text
校验分配头状态为 ALLOCATED
  -> 读取分配明细
  -> 根据源货位查巷道
  -> 执行下架策略 PolicyType.Picking
  -> 按容器分组创建拣货任务
  -> 回写 TaskHeaderId 到分配明细
  -> 更新分配头状态 PICKING
```

按容器分组创建任务这一点很关键。它意味着一次出库单可能生成多个任务，任务粒度和容器绑定，而不是和订单头一一对应。

### 12.9.3 拣货完成: 任务完成后的库存移动

`TaskHeaderService.PostTaskCompletionAsync` 对拣货任务会调用：

```csharp
_inventoryContract.MoveInventoryLocationAsync(...)
```

`InventoryContract.MoveInventoryLocationAsync` 做的是库存位置移动，不是扣减数量：

1. 找到容器在原货位的库存头。
2. 更新库存头的 `LocationId`、`LocationCode`。
3. 按库存明细写 `InvMovement` 流水，方向为不变。

所以拣货完成不等于库存扣减。库存扣减要看后续业务动作，比如复核、出库确认或无单据拣货扣减。

## 12.10 策略引擎在业务链路里的位置

入库、出库、任务链路都没有把“怎么选库区、怎么选库存、怎么选货位”硬编码在 Service 里，而是通过策略引擎。

| 场景 | 调用点 | 策略类型 |
| --- | --- | --- |
| 上架请求 | `InboundContainerBindService.ExecutePutawayStrategyAsync` | `PolicyType.Putaway` |
| 出库分配 | `OutboundAllocationService.ExecuteOutboundAllocationStrategyAsync` | `PolicyType.InventoryAllocation` |
| 生成拣货任务 | `OutboundAllocationService.ExecutePickingStrategyAsync` | `PolicyType.Picking` |
| 上架货位分配 | `TaskHeaderService.ExecuteLocationAllocationStrategyAsync` | `PolicyType.LocationAllocation` |

策略引擎的好处是：流程 Service 只关心“我要一个结果”，不直接关心结果如何计算。

但是排查时也要记住，如果接口提示“无可用库存”“未找到出库口”“未配置策略链”，问题可能不在 Service，而在策略配置、策略输入参数或策略查询服务。

## 12.11 可配置校验的执行位置

`InboundContainerBindService.ContainerBindAsync` 上有多条：

```csharp
[ConfigValidation(ValidatorCodes.BIND_DATA_NOT_EMPTY)]
[ConfigValidation(ValidatorCodes.BIND_QUANTITY)]
[ConfigValidation(ValidatorCodes.BATCH_NO_REQUIRED)]
[ConfigValidation(ValidatorCodes.EXPIRY_DATE_REQUIRED)]
[ConfigValidation(ValidatorCodes.MIXED_MATERIAL)]
[ConfigValidation(ValidatorCodes.MIXED_BATCH)]
```

这些校验不是方法体里手动调用，而是通过 `ConfigValidationInterceptor` 执行。

这带来两个开发要求。

1. 方法必须通过被 Autofac 代理的接口调用。
2. Service 注册不能关闭拦截器。

如果你发现 Validator 没执行，优先检查：

| 检查点 | 说明 |
| --- | --- |
| 是否注入接口 | Controller 或调用方应注入 `IInboundContainerBindService` |
| 是否写了 `[RegisteredService]` | 没注册就没有代理 |
| 是否 `WithoutInterceptor = true` | 开了就不会走校验拦截器 |
| 方法是否在接口上可见 | 代理调用的是接口暴露的方法 |
| 是否类内部自调用 | 自调用不会经过接口代理 |

## 12.12 并发与幂等: WMS 后端最容易出问题的地方

当前源码里已经有几个值得沿用的并发模式。

### 12.12.1 更新锁加状态复核

组盘：

```text
容器记录 With(SqlWith.UpdLock)
  -> 校验无活跃组盘、无库存、无活跃任务
```

出库分配：

```text
出库单 With(SqlWith.UpdLock)
  -> 复核状态不是 RELEASED
```

WCS 任务完成：

```text
任务 With(SqlWith.UpdLock)
  -> 复核状态仍是 PENDING 或 IN_PROGRESS
  -> 已完成则幂等返回成功
```

这种模式适合“先判断再写入”的流程，尤其是容器、库存、任务、订单状态这类共享资源。

### 12.12.2 条件更新保证库存锁定原子性

库存锁定使用条件更新：

```text
LockedQty = LockedQty + lockQty
WHERE Qty - LockedQty >= lockQty
```

这是比“查出来判断再 Update”更可靠的方式。后续写库存相关能力时，应优先考虑这种数据库原子条件，而不是只靠 C# 里的 if 判断。

### 12.12.3 WCS 回调必须幂等

WCS、PDA、外部接口都可能重复调用。任务完成接口的做法是：

1. 先锁任务。
2. 再判断任务是否还可完成。
3. 如果已经处理过，返回成功而不是重复执行副作用。

这类接口不要简单抛“状态不允许”，否则外部系统重试会变成业务噪音。

## 12.13 修改业务时应该去哪一层

| 需求 | 优先修改位置 | 不建议 |
| --- | --- | --- |
| 新增普通维护页 | Entity、Interface、Service、Controller | 复制一整套 CRUD 代码 |
| 新增列表过滤 | Service 重写 `BuildQueryExpression` 或 `ApplyAdditionalQuery` | Controller 拼 SQL |
| 新增写入前校验 | Service 的 `BeforeCreateAsync`、`BeforeUpdateAsync`，或 Validator | 前端校验后就信任 |
| 新增跨模块动作 | 被调用模块定义 Contract，调用方注入 Contract | 直接引用对方 `Services` |
| 修改状态流转 | Config 状态配置、状态校验 Contract、领域方法 | 到处直接赋值状态 |
| 修改库存锁定规则 | `InventoryContract` 或策略链 | 出库 Service 里绕过库存 Contract |
| 修改上架/下架/分配策略 | Algorithms 策略配置和策略实现 | 在流程 Service 里硬编码 |
| 修改导出字段 | `ExportAsync`、列配置、ExtData 展开逻辑 | 前端自己拼下载数据 |

一个简单判断：如果这个能力将来会被别的模块复用，就不要只写在当前 Controller 里。

## 12.14 排查路径: 从现象反推源码位置

| 现象 | 第一检查点 | 第二检查点 |
| --- | --- | --- |
| Swagger 看不到接口 | Controller 所在程序集是否被 `ApplicationPartManager` 加入 | Server 是否引用该模块项目 |
| DI 注入失败 | Service 是否有 `[RegisteredService(ServiceType = typeof(...))]` | 接口是否在构造函数中注入正确 |
| AOP 校验没执行 | 是否通过接口调用 | 是否 `WithoutInterceptor = true` 或类内部自调用 |
| ExtData 没保存 | `Program.cs` 是否启用 `EnableBuffering` | 请求体是否真的有 `extDataRaw` |
| ExtData 回显缺字段 | `ExtData` JSON 是否有效 | 列表接口是否需要前端或导出逻辑展开 |
| 新增/更新后从表没保存 | 实体导航属性是否有 `[Navigate]` | 是否传入 `IDetailSaveService` |
| 事务没有回滚 | 是否在事务内早返回 | catch 是否调用 `RollbackAsync` 后再返回或抛出 |
| 出库重复分配 | 是否对订单加锁并复核状态 | 库存锁定是否走 `IInventoryContract.LockInventoryAsync` |
| WCS 重复回调产生副作用 | 是否对任务加锁 | 已处理状态是否幂等返回 |
| 状态不能推进 | `IDocumentStatusValidatorContract` 配置 | 当前状态和目标状态是否允许流转 |
| 策略无结果 | 策略链配置 | `PolicyContext` 输入参数是否完整 |

## 12.16 新增一个流程接口的落地模板

下面这个模板适合入库、出库、库存、任务这类多表多模块流程。

```text
1. 判断归属模块
2. 定义请求 DTO
3. 在本模块 Interface 暴露流程方法
4. Service 方法通过接口被 Controller 调用
5. 如需跨模块能力，先检查是否已有 Contract
6. 没有 Contract 时，由能力拥有方定义 Contract 接口和实现
7. Service 开启 UnitOfWork 事务
8. 对共享资源加锁并复核状态
9. 执行配置校验或状态校验
10. 调用策略引擎取得决策结果
11. 写本模块表
12. 调用其他模块 Contract 写对方资源
13. 提交事务
14. 失败时回滚，并返回可读错误
15. 为重复请求设计幂等行为
```

伪代码结构：

```csharp
public async Task<ServiceResult> DoBusinessAsync(RequestDto dto)
{
    if (dto == null)
        return ServiceResult.Fail("请求不能为空");

    try
    {
        await _unitOfWork.BeginTransactionAsync();

        var locked = await _unitOfWork.DbContext.Db.Queryable<MyEntity>()
            .Where(x => x.Id == dto.Id)
            .With(SqlWith.UpdLock)
            .FirstAsync();

        if (locked == null)
        {
            await _unitOfWork.RollbackAsync();
            return ServiceResult.Fail("数据不存在");
        }

        if (!CanProcess(locked))
        {
            await _unitOfWork.RollbackAsync();
            return ServiceResult.Ok("已处理，无需重复提交");
        }

        await _statusValidator.ValidateTransitionAsync(...);

        var policyResult = await _policyEngine.ExecuteAsync(...);
        if (!policyResult.IsSuccess)
        {
            await _unitOfWork.RollbackAsync();
            return ServiceResult.Fail(policyResult.ErrorMessage);
        }

        await _repository.UpdateAsync(locked);
        await _otherContract.DoSomethingAsync(...);

        await _unitOfWork.CommitAsync();
        return ServiceResult.Ok("处理成功");
    }
    catch (Exception ex)
    {
        await _unitOfWork.RollbackAsync();
        return ServiceResult.Fail(ex.Message);
    }
}
```

这个模板的重点不是代码格式，而是几个原则：先锁关键资源，再复核状态；流程 Service 控制事务；跨模块只走 Contract；失败路径明确回滚；外部回调要考虑幂等。
