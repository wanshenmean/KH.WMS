## 2. 核心模块关系

| 业务域 | 后端模块 | 关键职责 |
| --- | --- | --- |
| 基础资料 | `BaseDataModule` | 物料、容器、客户、供应商、单位 |
| 仓储基础 | `WarehouseModule` | 仓库、库区、库位、站台、接驳点 |
| 配置 | `ConfigModule` | 单据、状态、扩展字段、编码、全局配置 |
| 入库 | `InboundModule` | 入库单、收货、质检、容器绑定 |
| 库存 | `InventoryModule` | 库存生成、查询、冻结、调整、盘点 |
| 出库 | `OutboundModule` | 出库单、波次、分配、发运 |
| 任务 | `TaskModule` | 上架、拣选、调拨、任务确认、WCS 日志 |
| 策略 | `KH.WMS.Algorithms` | 上架、货位、库存分配、下架、补货 |

## 3. 入库流程

典型链路：

```text
创建入库单
  -> 收货登记
  -> 容器绑定
  -> 生成上架任务
  -> 上架确认
  -> 生成库存
  -> 推进入库单状态
```

涉及模块：

| 阶段 | 模块 | 典型职责 |
| --- | --- | --- |
| 创建/维护入库单 | `InboundModule` | 入库单头、明细、状态 |
| 读取物料/容器 | `BaseDataModule` | 物料属性、容器状态 |
| 扩展字段/状态规则 | `ConfigModule` | 表单字段、状态流转 |
| 创建上架任务 | `TaskModule` | 任务头、任务行 |
| 推荐库位 | `WarehouseModule` + `Algorithms` | 候选库位、上架策略 |
| 生成库存 | `InventoryModule` | 库存头、库存明细 |

关键 Contract：

- `IContainerContract`
- `ITaskContract`
- `IInventoryContract`
- `ILocationContract`
- `IInboundOrderContract`

注意事项：

- 收货和上架通常不是同一个业务动作，不要在收货时直接生成最终库存。
- 上架完成后必须回写入库单进度。
- 容器状态、库位状态、库存状态需要保持一致。

入库流程的关键数据对象：

| 对象 | 含义 | 典型位置 |
| --- | --- | --- |
| 入库单头 | 本次入库业务主单 | `Entities/Inbound/InboundOrder.cs` |
| 入库单行 | 物料、数量、批次明细 | `Entities/Inbound/InboundOrderLine.cs` |
| 容器绑定 | 容器与物料关系 | `InboundContainerBind*` |
| 上架任务 | 执行动作 | `Entities/Task/TaskHeader.cs` |
| 库存头/明细 | 最终库存结果 | `Entities/Inventory` |

## 4. 出库流程

典型链路：

```text
创建出库单
  -> 波次/分配
  -> 库存锁定
  -> 创建拣选任务
  -> 拣选确认
  -> 扣减库存
  -> 复核/发运
  -> 推进出库单状态
```

涉及模块：

| 阶段 | 模块 | 典型职责 |
| --- | --- | --- |
| 出库单 | `OutboundModule` | 订单头、订单行 |
| 波次 | `OutboundModule` | 批量组织出库任务 |
| 库存分配 | `InventoryModule` + `Algorithms` | FIFO/FEFO/批次等规则 |
| 拣选任务 | `TaskModule` | 创建、确认、关闭 |
| 扣减库存 | `InventoryModule` | 库存扣减、锁定释放 |
| 发运 | `OutboundModule` | 物流信息、状态推进 |

关键 Contract：

- `IInventoryContract`
- `ITaskContract`
- `IOutboundContract`
- `ILocationContract`

注意事项：

- 分配阶段通常先锁库存，拣选确认后再扣减库存。
- 出库策略不要写死在 `OutboundService` 中，应接入 `KH.WMS.Algorithms`。
- 拣选任务完成后必须推进出库单/波次状态。

出库流程的关键风险：

- 库存分配和库存扣减时点混淆。
- 锁库后异常未释放。
- 任务完成但出库单未推进。
- 波次与订单状态不一致。
- 策略无库存时没有清晰提示。

## 5. 库存流程

库存能力包括：

- 库存生成。
- 库存查询。
- 库存移动。
- 库存冻结/解冻。
- 库存调整。
- 库存盘点。
- 库存快照。
- 库存预警。

涉及实体目录：

- `KH.WMS/Entities/KH.WMS.Entities/Inventory`

涉及模块：

- `InventoryModule`
- `TaskModule`
- `WarehouseModule`
- `InboundModule`
- `OutboundModule`

注意事项：

- 其他模块不要直接修改库存表，应通过 `IInventoryContract` 或库存模块服务。
- 任何影响库存数量的动作都要明确业务来源和 TraceId。
- 冻结、锁定、扣减是不同语义，不能混用。

库存变更建议记录：

- 来源单据号。
- 来源任务号。
- 容器号。
- 库位。
- 物料、批次、数量。
- 操作人。
- TraceId 或 BusinessId。

## 6. 任务流程

任务中心负责把业务单据转成可执行动作：

- 入库上架任务。
- 出库拣选任务。
- 调拨任务。
- 临时任务。
- WCS 任务日志。

典型链路：

```text
业务单据产生需求
  -> TaskModule 创建任务
  -> PDA/WCS 执行
  -> TaskConfirm 确认
  -> 回写业务模块和库存模块
```

注意事项：

- 任务确认不是单纯改任务状态，通常伴随库存、容器、库位、单据状态变化。
- 任务并发确认需要考虑幂等和锁。
- WCS 通信失败不应让核心库存事务处于不可控状态。

任务确认的事务建议：

```text
校验任务状态
  -> 校验库位/容器/库存
  -> 写任务确认
  -> 更新库存或库位
  -> 推进业务单据
  -> 写必要日志
```

外部系统调用，例如 WCS/AGV 下发，应谨慎放入事务中。通常建议核心数据库事务先保证一致性，再处理外部通信和补偿。

## 7. 仓储基础流程

仓储基础是空间和设备底座：

- 仓库。
- 库区。
- 巷道。
- 库位。
- 逻辑区。
- 站台。
- 输送线。
- 接驳点。

常见业务用途：

- 入库推荐库位。
- 出库分区。
- 站台接驳。
- 库位占用/释放。
- 逻辑区策略分组。

注意事项：

- 物理库区、巷道、逻辑区概念不要混淆。
- 库位状态变化应有依据，不能绕过仓储模块随意修改。
- 策略算法应通过仓储查询服务或 `ILocationContract` 获取空间数据。

## 8. 业务流程中的数据一致性

跨模块业务最常见的问题不是单个接口失败，而是状态不一致：

| 不一致场景 | 可能原因 | 检查方向 |
| --- | --- | --- |
| 入库单已收货但无任务 | 创建任务失败未回滚 | Inbound + Task 事务边界 |
| 任务已完成但无库存 | 库存生成失败 | TaskConfirm + InventoryContract |
| 库存扣减但出库单未完成 | 状态推进失败 | Inventory + Outbound |
| 库位占用但库存不在该库位 | 库位状态和库存未同步 | Warehouse + Inventory |
| 容器状态异常 | 容器状态更新漏调用 | BaseData Contract |

培训时要求成员能说出每条流程的“主事务”在哪里，以及失败后应该回滚哪些数据。

## 9. 培训案例：PDA 收货到上架完成

建议课堂演练：

1. PDA 页面提交收货或容器绑定。
2. InboundModule 校验入库单和物料。
3. BaseDataModule 处理容器。
4. ConfigModule 判断扩展字段和状态流转。
5. TaskModule 创建上架任务。
6. WarehouseModule 提供目标库位。
7. TaskConfirm 确认上架。
8. InventoryModule 生成库存。
9. InboundModule 推进入库单状态。
10. 前端页面刷新看到状态变化。

每一步都要求指出：

- 调用哪个模块。
- 是否通过 Contract。
- 是否需要事务。
- 失败时返回什么错误。
- 前端如何展示。

## 10. 流程评审清单

- 是否识别主单、明细、任务、库存、库位、容器之间的关系？
- 是否明确状态流转？
- 是否有跨模块 Contract？
- 是否有事务边界？
- 是否有幂等或重复提交处理？
- 是否有日志和 TraceId？
- 是否有前端状态刷新和错误提示？

## 11. 课后检查

- 能说出入库、出库、库存、任务之间的先后关系。
- 能指出一个流程中哪些模块负责主流程，哪些模块提供能力。
- 能判断库存数量变更应通过哪个模块。
- 能说明为什么任务确认是多个模块协作点。
