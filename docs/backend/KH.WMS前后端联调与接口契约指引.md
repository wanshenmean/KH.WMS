## 1. 阅读路径

1. 后端先读 `KH.WMS后端接口路由与前端联调清单.md`。
2. 前端先读 `KH.WMS前端请求封装与接口开发指引.md`。
3. 联调时使用本文统一检查。

## 2. 联调前准备

后端：

- `dotnet build` 通过。
- 后端服务启动。
- Swagger 可访问。
- 登录接口可用。
- 目标接口能在 Swagger 中验证。

前端：

- `npm install` 完成。
- `npm run dev` 成功。
- Vite `/api` 代理指向 `http://localhost:9291`。
- 登录后能拿到 token。

联调前必须先确认“接口本身可用”，再看“页面接入是否正确”。如果 Swagger 都不能稳定成功，前端页面调试只会消耗时间。

## 3. 联调角色分工

| 角色 | 负责内容 |
| --- | --- |
| 后端 | 路由、DTO、业务校验、权限、事务、日志、TraceId |
| 前端 | API 封装、页面交互、错误提示、权限显示、字典/扩展字段渲染 |
| 测试/验收 | 主流程、异常流程、权限流程、数据一致性 |

接口契约确认顺序：

1. 确认 URL、HTTP Method、请求体。
2. 确认成功响应 `data` 结构。
3. 确认失败响应 `code/message/traceId`。
4. 确认分页、排序、过滤规则。
5. 确认字典、权限、扩展字段是否参与页面渲染。
6. 确认异常场景提示文案由谁提供。

## 4. 路由与 API 对应

后端：

```csharp
[Route("api/material")]
public class MaterialController(...)
```

前端：

```js
request.get('/api/material/...')
```

约定：

- 后端 Controller 路由决定前端请求前缀。
- 前端接口放在对应 `src/api/{module}.js`。
- 页面不直接拼复杂请求路径。

建议建立一张接口对照表：

| 页面动作 | 前端 API 方法 | 后端路由 | Method | 备注 |
| --- | --- | --- | --- | --- |
| 查询列表 | `getBrandPage` | `/api/basedata/brand/pagelist` | POST | 分页查询 |
| 查看详情 | `getBrandDetail` | `/api/basedata/brand/{id}` | GET | ID 路径参数 |
| 新增 | `createBrand` | `/api/basedata/brand/create` | POST | DTO 校验 |
| 编辑 | `updateBrand` | `/api/basedata/brand/update` | POST/PUT | 以实际后端为准 |
| 删除 | `deleteBrand` | `/api/basedata/brand/delete/{id}` | DELETE/POST | 以实际后端为准 |

## 5. ApiResponse 判断

后端统一响应字段：

- `code`
- `message`
- `data`
- `traceId`
- `timestamp`

前端成功判断：

```js
res.code === 200 || res.code === 0
```

失败处理：

- 优先展示后端 `message`。
- 需要排查时复制 `traceId`。
- 字段级错误由页面表单或统一错误处理展示。

前端页面只关心业务成功与失败，不应该每个页面都重新发明响应判断。培训时要强调：响应结构统一是为了让页面代码少写分支，也为了排错时能拿到同一种证据。

## 6. 分页查询契约

前端 `buildPageQuery` 发送：

```js
{
  pageIndex,
  pageSize,
  sortConditions,
  filters
}
```

后端分页接口：

```text
POST /api/{module}/pagelist
```

约定：

- 搜索框空值不传。
- select 类型默认 equals。
- input 类型默认 contains。
- 多选使用 in。
- 表头筛选追加到 filters。

联调时必须拿真实查询条件验证：

| 场景 | 检查点 |
| --- | --- |
| 空查询 | 不应发送空字符串过滤条件 |
| 模糊查询 | input 是否转成 contains |
| 状态筛选 | select 是否转成 equals |
| 多选筛选 | 是否转成 in |
| 表头排序 | 字段名和方向是否正确 |

## 7. 动态表单与扩展字段

后端：

- 提供 `/form-config`。
- 从 ConfigModule 扩展字段能力构建表单列。

前端：

- 调用 `formConfig()`。
- 使用 `useExtFields.js` 或表单配置渲染扩展字段。

约定：

- 没有扩展字段时返回空数组或默认配置。
- 字段类型、必填、选项、显示名称要稳定。
- 前端不要为每个模块自定义一套完全不同的动态字段结构。

扩展字段联调特别关注：

- 新增、编辑、详情是否使用同一套字段配置。
- 必填校验前后端是否一致。
- 字段值提交结构是否和后端保存结构一致。
- 字段删除或停用后旧数据如何展示。
- `/form-config` 失败时页面是否有兜底。

## 8. 字典契约

后端：

- SystemModule 维护字典。
- 提供按字典类型查询接口。

前端：

- `dict` Store 缓存字典。
- 页面通过字典类型获取下拉项。

约定：

- 前端不写死通用状态文案。
- 字典修改后需要刷新缓存。
- 字典只承载展示/枚举，不承载复杂业务规则。

字典联调不要只看下拉能不能显示，还要看：

| 场景 | 检查 |
| --- | --- |
| 列表展示 | 值能否解析成标签 |
| 表单编辑 | 选项是否启用、排序是否正确 |
| 字典变更 | 缓存是否刷新 |
| 未知值 | 是否有兜底展示 |

## 9. 权限契约

后端返回权限树，前端消费：

- 菜单：`permissionCode`、`path`、`component`、`menuType`。
- 按钮：`buttons[].permKey`。
- 可见性：`isVisible`。
- 启用状态：`status`。

前端动态路由：

```text
component = "system/user"
-> /src/views/system/user.vue
```

约定：

- 后端菜单 component 不带 `.vue`。
- 前端按钮隐藏不能替代后端权限校验。
- 权限变更后需要重新加载权限或重新登录。

权限联调必须同时验证：

1. 有权限用户：菜单可见、按钮可见、接口可调用。
2. 无按钮权限用户：按钮不可见，直接调接口也失败。
3. 无菜单权限用户：菜单不可见，直接访问 URL 被拦截或拒绝。
4. 权限变更后：刷新权限或重新登录后生效。

## 10. TraceId 排错流程

1. 前端请求失败，记录 `traceId`。
2. 后端按 `CorrelationId` 或 `TraceId` 查日志。
3. 查看请求路径、用户、业务 ID、异常堆栈。
4. 定位 Controller、Service、SQL 或策略。

没有 `traceId` 时，也要记录请求时间、请求路径、用户、请求体、响应状态码，后端可按时间窗口和用户定位日志。

## 11. 联调问题记录模板

每个联调问题按下面格式记录，避免“页面不行”“接口有问题”这种无法定位的描述：

| 字段 | 示例 |
| --- | --- |
| 问题标题 | 品牌分页查询状态筛选无效 |
| 页面路径 | `/basedata/brand` |
| 前端 API | `getBrandPage` |
| 后端路由 | `POST /api/basedata/brand/pagelist` |
| 请求参数 | `status = 1` |
| 期望结果 | 只返回启用品牌 |
| 实际结果 | 返回全部品牌 |
| traceId | 从响应或日志复制 |
| 初步判断 | filters 操作符或后端字段映射问题 |
| 负责人 | 前端/后端/共同 |

## 12. 联调完成标准

- Swagger 验证通过。
- 前端页面功能通过。
- 成功/失败提示正确。
- 权限控制正确。
- 分页、筛选、排序正确。
- TraceId 可用于排错。
- 无需前端硬编码后端状态文案。

## 13. 课后检查

- 能从后端路由找到前端 API 文件。
- 能解释分页查询结构。
- 能说明扩展字段、字典、权限如何前后端配合。
- 能按 TraceId 进行一次完整排错。
- 能写出一条完整的联调问题记录。
