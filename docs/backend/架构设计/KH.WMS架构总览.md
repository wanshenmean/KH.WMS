# KH.WMS 架构总览

这篇文档回答三个问题：KH.WMS 的前后端为什么这样组织、这种组织方式解决了什么问题、开发新功能时怎样不破坏现有边界。

如果只想先建立全局认知，读完本页即可。准备落代码时，再继续阅读[前端架构设计思路](./KH.WMS前端架构设计思路.md)或[后端架构设计思路](./KH.WMS后端架构设计思路.md)。

## 1. 架构首先服务什么

KH.WMS 不是一个只有几张维护表的后台系统。它同时要处理基础资料、入库、库存、出库、任务、仓库结构、策略算法、设备协作和权限配置。架构选择优先保护以下质量目标：

| 质量目标 | WMS 中的实际问题 | 架构回应 |
| --- | --- | --- |
| 数据一致性 | 单据、库存、任务、容器状态常在一个动作中联动 | 业务流程由后端 Service 编排，事务边界集中在服务层 |
| 可追踪性 | 现场问题必须知道哪个请求、哪条规则、哪一步失败 | 统一响应、异常处理、TraceId、日志和性能观测集中在后端底座 |
| 可配置性 | 不同仓库的字段、状态、编码和上架拣选规则不同 | Config 与 Algorithms 独立为配置、状态和策略底座 |
| 交付效率 | 大量页面和接口属于结构相似的 CRUD | 前端 KhPage 组件体系与后端 CRUD 基类提供标准路径 |
| 权限一致性 | 菜单、路由、按钮和接口权限不能各维护一套 | 后端下发权限树，前端统一生成菜单、动态路由和按钮权限 |
| 模块可维护性 | 入库、库存、任务互相协作，但不能互相侵入实现 | 后端按业务模块拆分，跨模块调用通过 Contract |
| 部署适应性 | 既要支持前后端同源，也可能分离部署 | 前端使用运行时 API 地址，后端可托管 SPA 并提供 history 回退 |

架构的目的不是增加层数，而是把容易变化的部分放到明确位置，把所有功能都必须遵守的规则集中起来。

## 2. 总体形态

当前系统采用“Vue 单页应用 + ASP.NET Core 模块化单体”的形态。

```text
浏览器
  ├─ PC 布局 / PDA 布局
  ├─ 动态路由、菜单、按钮权限
  ├─ 页面与 Kh 组件体系
  └─ API / request.js
           │ HTTP + JSON + Token
           ▼
KH.WMS.Server
  ├─ HTTP 管道、鉴权、异常、TraceId、Swagger
  ├─ Modules.* 业务模块
  │    ├─ Controller
  │    ├─ Service / Validation
  │    └─ Contract 实现
  ├─ Core 通用技术底座
  ├─ Entities 持久化模型
  ├─ Contracts 跨模块契约
  ├─ Config 配置与状态底座
  └─ Algorithms 策略算法底座
           │
           ▼
     业务库 / 配置库 / 外部设备与系统
```

### 2.1 为什么前端是单页应用

仓库操作页面需要频繁查询、编辑、弹窗、表格联动和权限切换。Vue Router 在浏览器内完成页面切换，Pinia 保存少量跨页面状态，组件体系统一复杂表格和表单交互，能够减少整页刷新和重复开发。

收益是交互一致、标准页面交付快、PC 与 PDA 可以共享请求和权限基础能力。代价是启动阶段要正确恢复用户与权限状态，动态路由、缓存状态和请求拦截器也会形成隐式执行链，因此必须保持入口集中并提供排查文档。

### 2.2 为什么后端是模块化单体

WMS 的核心流程经常跨越多个业务边界，并要求一次操作内保持一致。例如“申请上架”可能同时涉及入库数据、策略计算、任务创建和容器状态。模块化单体让这些能力仍处于一个进程和事务环境中，避免过早引入分布式事务、消息最终一致性和多服务部署成本。

同时，代码不是堆在一个项目里：入库、库存、出库、任务等仍按业务模块隔离，跨模块通过 Contract 协作。这样既保留单体的一致性和运维简单性，也避免业务实现无边界互相引用。

当前实现不是微服务，也没有通过 `ModuleBase` 驱动业务模块生命周期。业务模块主要依靠启动项目引用、程序集扫描、MVC `ApplicationPart` 和服务注册特性接入。`KH.WMS.Core.Modularity` 中的能力属于预留底座，不能当成已经启用的主链路。

## 3. 前后端职责边界

| 决策 | 前端负责 | 后端负责 | 为什么这样分 |
| --- | --- | --- | --- |
| 页面交互 | 展示、输入、即时交互校验、加载状态 | 最终业务校验 | 浏览器校验改善体验，但不能成为可信业务边界 |
| 权限 | 控制菜单、路由和按钮是否展示 | 验证调用者是否有接口权限 | 前端控制体验，后端控制安全 |
| 数据查询 | 组织查询条件、分页和排序请求 | 解释过滤条件并访问数据 | 数据规模、索引和访问规则属于服务端 |
| 业务流程 | 收集操作意图并展示结果 | 状态流转、事务、跨模块编排 | 关键规则必须只有一个可信实现 |
| 接口契约 | 按约定发送字段并处理统一响应 | 定义 DTO、路由、错误码和响应形状 | 契约稳定才能独立开发与联调 |
| 配置与策略 | 展示配置结果和可选项 | 解析配置、选择策略、执行规则 | 防止规则散落在页面并被绕过 |

一个简单判断方法：刷新页面、换一个终端或直接调用 API 后仍必须成立的规则，不能只写在前端。

## 4. 一次请求怎样穿过系统

以标准资料维护页为例：

```text
后端权限菜单 component
  -> router/index.js 匹配 src/views 下的页面
  -> 页面配置 KhPage 的 module、字段和权限前缀
  -> KhPage 组合 KhForm、KhTable、KhDialog
  -> useCrudApi(module) 生成标准端点
  -> request.js 注入 Token、处理 Loading 与错误
  -> /api/<module>/pagelist 或 create/update/delete
  -> Controller 接收 HTTP 请求
  -> Service 执行业务规则
  -> Repository / UnitOfWork 查询或写入
  -> ApiResponse + TraceId 返回
  -> request.js 统一处理后交给页面刷新状态
```

这条链路把变化分成三类：

- 页面字段和交互变化，主要留在 `views` 与页面局部组件。
- 可复用的 CRUD、权限、请求和响应规则，进入前后端公共底座。
- 状态流转、事务、跨模块协作和策略选择，进入后端业务层。

流程型接口会在 Service 中增加 Contract、Config 或 Algorithms 调用，但前端仍只表达“用户要做什么”，不承担最终流程编排。

## 5. 核心架构决策

| 问题 | 当前选择 | 主要收益 | 主要代价与边界 |
| --- | --- | --- | --- |
| 大量相似维护页 | KhPage 配置驱动页面 + 后端 CRUD 基类 | 交付快、交互与接口一致 | 非标准流程不能硬塞进通用抽象 |
| 菜单和权限经常变化 | 后端权限树驱动前端动态路由与按钮权限 | 权限来源统一，角色配置即时生效 | 刷新时必须先恢复用户和权限再进入页面 |
| 全局请求行为 | 单一 Axios 实例与拦截器 | Token、续期、错误和 Loading 统一 | 拦截器过重会让失败链路难理解 |
| 跨页面状态 | Pinia 只保存用户、权限、字典、应用与连接状态 | 生命周期明确、避免重复请求 | 页面局部数据进入 Store 会造成陈旧状态 |
| 业务代码组织 | 按业务能力拆分 `Modules.*` | 团队可以按领域维护，降低影响面 | 模块边界需要 Contract 和评审约束维护 |
| 跨模块调用 | 调用方依赖 `KH.WMS.Contracts` 接口 | 避免模块实现互相引用和循环依赖 | 契约需要稳定，不能暴露对方内部模型 |
| 横切能力 | Core 中的 DI、AOP、异常、响应、日志和事务 | 业务代码聚焦规则，行为一致 | 自动扫描和拦截属于隐式机制，需要可观测和文档 |
| 仓库差异 | Config + Algorithms 配置和策略底座 | 规则可替换，减少仓库专属分支 | 不是所有简单规则都值得抽成策略 |
| 部署方式 | 运行时 `config.js`，后端可托管 SPA | 同一构建可切换 API 地址，同源部署简单 | 分离部署仍需正确配置 CORS 与 history 回退 |

## 6. 这样设计带来的整体收益

### 6.1 更快交付标准需求

标准 CRUD 从前端配置、统一组件到后端基类形成一条“铺好的路”。开发者只补业务字段、必要校验和差异化行为，不需要为每个页面重新决定分页结构、按钮权限、响应格式和异常处理。

### 6.2 变化被限制在正确范围

页面变化通常不会触及事务；某个业务模块变化不需要了解其他模块实现；Token 或响应规范变化可以在请求层和后端底座统一处理。影响范围可预测后，代码评审和回归测试都会更聚焦。

### 6.3 现场问题更容易定位

动态路由、统一请求、Controller、Service、事务和 TraceId 构成清晰检查顺序。开发者可以判断问题是页面映射、权限、接口契约、业务校验还是数据写入，而不是从任意文件开始猜。

### 6.4 为未来拆分保留边界

模块化单体不等于永远不能拆服务。稳定的 Contract、清晰的数据归属和禁止直接引用实现，可以让未来有必要时更容易识别服务边界。但当前没有明确的独立扩缩容或部署需求时，不应为“可能拆分”提前承担分布式复杂度。

## 7. 代价、例外和风险

### 7.1 抽象会隐藏执行细节

KhPage、CRUD 基类、自动注册和 AOP 都能减少重复代码，但也让一部分行为不是从当前文件直接看到。解决方式不是放弃抽象，而是控制抽象职责、保留扩展钩子、记录真实执行链，并让日志能指出请求和事务结果。

### 7.2 标准路径不能覆盖所有业务

主从表保存、波次、上架、拣选、库存移动等流程常有专用 DTO、状态机和事务边界。它们可以复用请求、表格、仓储和响应底座，但不应该为了形式统一而伪装成普通 CRUD。

### 7.3 模块边界不会自动保持

项目目录只是边界的外壳。直接引用其他模块 Service、复制配置逻辑、共享可变 DTO 或在 Controller 写事务，都会逐步破坏边界。Contract 和代码评审规则必须同时存在。

### 7.4 配置化也有成本

配置和策略适合“因仓库、客户或场景而变化”的规则。稳定、简单且只有一个实现的判断直接写清楚通常更好。过度配置会把可读的代码变成难追踪的数据驱动流程。

## 8. 什么时候应该走例外路径

| 场景 | 推荐做法 |
| --- | --- |
| 普通单表维护 | 优先使用 KhPage、`useCrudApi`、`CrudController` 与 `CrudService` |
| 主从表或长业务流程 | 自定义页面交互和专用 DTO，由 Service 明确编排事务 |
| 跨模块读取或写入 | 定义最小 Contract，由数据所属模块实现 |
| 仓库可配置决策 | 使用 Config 或 Algorithms 的抽象，不在页面和业务 Service 散落仓库编号判断 |
| 页面临时状态 | 留在组件或 composable，不进入全局 Store |
| 高频实时推送 | 复用 WebSocket Store 的连接管理，按主题隔离业务消息 |
| 独立扩缩容、强隔离或不同发布节奏成为刚需 | 先以数据所有权和 Contract 评估服务拆分，不直接复制模块为微服务 |

## 9. 架构失控的早期信号

- 一个页面同时直接调用许多互不相关的 API，并自己拼装关键业务状态。
- 页面局部表单进入全局 Store，离开页面后仍残留旧数据。
- 新页面复制 KhTable、分页、权限和错误处理，而不是复用或明确说明例外。
- Controller 直接注入多个 Repository、开启事务并编排跨模块流程。
- 一个业务模块直接引用另一个业务模块的 Service 项目。
- 为解决循环依赖，把越来越多业务实现移进 Core 或 Contracts。
- 仓库差异写成大量 `warehouseId` 分支，且相同规则在前后端各有一份。
- 捕获异常后返回“成功”，导致统一异常、TraceId 和事务机制失去作用。

出现这些信号时，应先确认职责和数据所有权，再决定是复用底座、增加扩展点、抽 Contract，还是保留一个清晰的专用实现。

## 10. 开发前的架构检查

开始一个功能前，至少回答以下问题：

1. 需求属于哪个业务模块，数据的最终所有者是谁？
2. 它是标准 CRUD、主从表，还是需要状态流转的业务流程？
3. 页面能否复用 KhPage；如果不能，具体差异是什么？
4. 哪些规则只影响交互，哪些规则必须由后端保证？
5. 是否跨模块；如果跨模块，调用方需要的最小 Contract 是什么？
6. 事务从哪个业务动作开始，到哪些写入全部成功才结束？
7. 哪些变化属于仓库配置或策略，是否应该进入 Config 或 Algorithms？
8. 失败后用户看到什么，日志和 TraceId 是否足够定位？
