# KH.WMS 前端架构设计思路

KH.WMS PC 前端的核心目标不是“把接口显示出来”，而是在大量资料维护、业务操作、权限菜单和现场终端页面之间，建立一条一致、可扩展、可排查的开发路径。

本文重点解释设计原因和边界。具体页面写法请结合[KH.WMS PC 前端开发指引 V3.0](/backend/KH.WMS前端开发指引%20V3.0)阅读。

## 1. 当前技术与目录结构

前端是 Vue 3 + Vite + Vue Router + Pinia + Element Plus 项目，主要入口如下：

```text
KH.WMS.Client/
  public/config.js          部署后可修改的 API 地址
  src/
    main.js                 应用装配入口
    api/                    业务 API
    components/             全局复用的 Kh 组件
    composables/            可组合状态与行为
    directives/             按钮权限等指令
    layouts/                PC 与 PDA 布局
    router/                 静态路由、动态路由和守卫
    stores/                 跨页面共享状态
    utils/                  请求、CRUD、扩展字段等基础能力
    views/                  按业务领域组织的页面
```

整体依赖方向应保持为：

```text
main.js
  -> router / stores / global plugins

layouts
  -> router view / navigation / shared application state

views
  -> page-local components / Kh components / composables / api

Kh components
  -> lower-level Kh components / Element Plus

api / crud
  -> request.js
  -> backend HTTP API
```

页面不应该绕过 `request.js` 自建 Axios 实例，通用组件也不应该反向依赖某个具体业务页面。

## 2. 前端架构要解决的问题

| 问题 | 架构目标 |
| --- | --- |
| 大量页面结构相似 | 用配置驱动组件减少重复表单、表格、弹窗和 CRUD 代码 |
| 菜单和权限由角色配置 | 让同一份权限数据驱动菜单、路由和按钮 |
| Token、续期和错误处理复杂 | 所有 HTTP 请求经过统一出口 |
| 用户、权限和字典跨页面共享 | 只把真正跨页面且有明确生命周期的数据放进 Pinia |
| PC 与 PDA 交互差异明显 | 共用应用与请求底座，使用独立布局和页面交互 |
| 部署环境不同 | API 地址在构建后仍可切换 |
| 新成员定位问题慢 | 让调用链和代码落点可以按层排查 |

这套设计优先选择“约定和一致性”，而不是让每个页面自由选择技术方案。

## 3. 应用装配为什么集中在 main.js

`src/main.js` 只负责应用级装配：创建 Vue 应用、安装 Pinia、Router 和 Element Plus，注册权限指令、图标及少量全局能力，最后挂载根组件。

### 解决什么问题

应用启动时必须确定插件顺序、全局指令和全局能力。如果这些注册散落在布局或业务页面中，页面刷新顺序不同就可能得到不同结果。

### 为什么这样设计

把装配入口集中后，业务页面只消费能力，不负责初始化框架。`router`、Store 和 UI 组件库都只有一个应用级实例。

### 获得的收益

- 启动顺序清晰。
- 全局能力不会被重复注册。
- 单元或 E2E 排查时容易判断应用是否完成装配。

### 代价与边界

`main.js` 不应成为“方便放全局变量”的文件。业务 API 全量挂载到 `globalProperties` 虽然兼容选项式写法，但新代码仍应优先显式导入需要的 API，避免依赖来源不可见。

## 4. 为什么区分布局、页面和组件

### 4.1 Layout 负责应用外壳

`PcLayout.vue` 与 `PdaLayout.vue` 负责导航、页面容器以及不同终端的整体交互骨架。

这样设计的原因是 PC 后台和 PDA 作业有不同的信息密度与操作方式，但它们仍共享登录、请求、权限和数据契约。独立 Layout 可以隔离交互差异，而不复制整个应用。

Layout 不应承载物料、入库或库存等业务流程，否则所有页面都会被迫依赖某个业务模块。

### 4.2 View 负责一个业务页面

`src/views` 按业务域组织页面，例如 `basedata`、`inbound`、`outbound`、`system` 和 `warehouse`。View 负责组合查询、表格、表单、按钮和页面局部状态。

页面局部弹窗和仅供当前页面使用的组件放在对应 `components/` 子目录。这样可以避免全局组件目录被一次性组件淹没，也让删除页面时能同时删除它的局部实现。

### 4.3 全局组件负责稳定复用能力

`src/components` 中的 Kh 组件不是普通页面片段，而是项目级交互协议：

- `KhPage` 组合标准查询、表格、CRUD 按钮和编辑弹窗。
- `KhTable` 统一分页、排序、筛选、选择和工具栏交互。
- `KhForm` 统一配置化字段渲染与校验。
- `KhDialog` 统一新增、编辑、查看等弹窗行为。

提升为全局组件前，应满足至少两个条件：多个业务页面稳定复用，并且输入输出可以脱离具体业务命名。否则保留为页面局部组件更容易维护。

## 5. 为什么采用 KhPage 配置驱动页面

标准资料页通常重复以下结构：查询条件、分页表格、新增编辑查看、删除确认、权限按钮、导出和状态切换。每个页面手写会产生大量几乎相同但细节不一致的代码。

```text
业务页面配置
  -> KhPage
      -> KhForm 查询
      -> KhTable 列表
      -> KhDialog 编辑/查看
      -> useCrudApi 标准接口
      -> permissionPrefix 按钮权限
```

### 为什么这样设计

KhPage 把稳定的页面骨架收敛起来，把变化留给 `module`、查询列、表格列、表单列、插槽、事件和自定义加载函数。页面声明“有哪些字段和差异”，不再重复实现整套生命周期。

### 获得的收益

- 新增标准页面时代码量少，培训路径统一。
- 查询、分页、权限、编辑和错误处理表现一致。
- 修复通用问题时可以一次覆盖多个页面。
- 前后端标准 CRUD 路由形成稳定契约。

### 付出的代价

- 执行行为分布在 KhPage、KhTable、KhDialog 和 CRUD 工厂中，新人需要理解调用链。
- 通用组件的改动影响面大，必须保持向后兼容。
- 复杂页面如果不断给 KhPage 增加只服务一个场景的属性，会使抽象失控。

### 什么时候不适用

以下场景应组合底层 Kh 组件或编写专用页面，而不是强行扩展 KhPage：

- 主从表需要一次提交复杂请求体。
- 波次、上架、拣选等流程具有多步骤状态和专用操作。
- 页面以看板、拓扑、拖拽或实时设备状态为主。
- 同一页面同时编排多个不同聚合的数据。

例外页面仍应复用 `request.js`、权限指令、消息提示和适合的 Kh 基础组件。

## 6. 动态路由和权限为什么由后端数据驱动

`src/router/index.js` 使用 `import.meta.glob('../views/**/*.vue')` 预加载可选页面模块，根据后端权限树中的 `component` 字段精确匹配 `src/views/<component>.vue`，再把路由注册到 PC Layout 下。

```text
登录 Token
  -> 获取用户信息
  -> permission Store 获取权限树
  -> 权限树生成菜单与 dynamicRoutes
  -> router 精确匹配 views 文件
  -> addRoute 注册受保护页面
  -> 路由 meta + permission Store 校验访问
  -> v-permission / KhPage 控制按钮
```

### 解决什么问题

角色权限会变化。如果静态路由、左侧菜单和按钮权限分别维护，三者很容易不一致。

### 为什么这样设计

后端权限树是配置来源，前端只负责把同一份数据转换为导航和交互状态。页面文件仍由前端构建产物决定，后端不能注入任意组件代码。

### 获得的收益

- 菜单、路由和按钮共享权限编码。
- 角色配置变化不要求修改静态前端路由表。
- 页面刷新时可从 Token 恢复用户与权限。
- `component` 与 `views` 路径形成可检查的明确约定。

### 代价与边界

- 首次进入受保护页面前必须完成用户和权限加载。
- 菜单 `component` 写错时会落到占位页，因此联调必须同时检查后端配置和前端文件。
- 前端隐藏按钮不等于安全控制；后端接口仍必须鉴权。
- 动态路由名称和清理逻辑必须防止重复注册和用户切换后的残留。

## 7. 为什么所有请求经过 request.js

`src/utils/request.js` 创建唯一 Axios 实例，并集中处理：

- `public/config.js` 提供的运行时 API 地址。
- Authorization Token 注入。
- 全局 Loading 计数。
- 响应头静默更新 Token。
- 401 刷新 Token 及并发请求排队。
- 402 License 页面跳转。
- 强制退出时清理用户、权限、字典和 WebSocket 状态。
- 统一错误提示和响应体返回。

### 为什么这样设计

这些行为属于所有 API 调用的传输协议，而不是某个业务页面的规则。集中后可以保证登录过期、并发续期和错误展示只有一个实现。

### 获得的收益

- API 文件只描述路由、方法和参数。
- Token 与错误处理不会在页面复制。
- 并发续期只发起一次刷新，其他请求等待结果。
- 退出登录能同步清理跨页面资源。

### 付出的代价

请求拦截器是一条全局隐式链。修改时必须覆盖成功、业务失败、401、402、网络错误和并发请求，不能只验证一个页面。

业务 API 不应在拦截器中判断物料、库存或单据状态。拦截器只处理传输层与应用级会话行为。

## 8. 为什么标准 CRUD 与业务 API 分开

`src/utils/crud.js` 根据模块名生成标准端点，并把 KhTable 的扁平查询条件转换为后端分页、排序和过滤结构。

```text
useCrudApi('material')
  -> /api/material/pagelist
  -> /api/material/create
  -> /api/material/update
  -> /api/material/delete/{id}
  -> /api/material/status/{id}
  -> /api/material/form-config
  -> /api/material/export
```

标准行为使用工厂可以消除重复。非标准动作放在 `src/api/<domain>.js`，例如审批、申请上架、确认任务或聚合查询，让方法名表达业务意图。

不要为了复用工厂而把流程接口伪装成 CRUD，也不要在页面中到处拼接 URL。判断标准是：端点是否遵循统一资源操作和统一参数结构。

## 9. Pinia 为什么只保存跨页面状态

当前 Store 主要承担以下职责：

| Store | 适合保存的状态 | 生命周期 |
| --- | --- | --- |
| `user` | Token、刷新 Token、当前用户 | 登录到退出 |
| `permission` | 权限树、路由权限、按钮权限、动态路由描述 | 当前用户与角色 |
| `dict` | 字典缓存和并发加载 Promise | 会话内，可按字典刷新 |
| `websocket` | 通知与设备连接、连接状态 | 登录会话或布局生命周期 |
| `app` | 真正影响多个页面的应用级 UI 状态 | 应用会话 |

### 为什么这样设计

跨页面状态需要在布局、路由守卫、页面和请求错误处理中共享。Pinia 提供明确的状态、getter 和 action，并能被 Vue 响应式消费。

### 收益

- 用户和权限恢复逻辑集中。
- 字典请求可以缓存并去重。
- 退出登录时能统一释放连接和缓存。
- 页面不需要通过多层 props 传递应用状态。

### 代价与不适用场景

Store 生命周期通常长于页面。查询表单、编辑草稿、当前弹窗和临时选中行默认留在页面；放进 Store 后容易在页面跳转或用户切换时残留。

只有多个不相邻组件确实需要共享、需要跨路由保留，或需要集中处理副作用时，才应新增 Store。单个页面内复用逻辑优先使用 composable。

## 10. 为什么保留运行时 config.js

Vite 环境变量通常在构建时固化，而 WMS 交付环境的 API 地址可能在安装后才确定。`public/config.js` 在打包后成为 `dist/config.js`，通过 `window.__APP_CONFIG__.API_BASE_URL` 为请求实例提供地址。

### 获得的收益

- 同一份前端构建可以部署到不同现场。
- 前后端同源时留空即可使用相对 `/api`。
- 分离部署时只修改配置文件，不需要重新编译。

### 代价与边界

- 配置文件必须先于应用脚本加载，并处理浏览器缓存。
- 它是公开运行时配置，不能存密码、密钥或其他秘密。
- 跨域地址仍要求后端正确配置 CORS。

## 11. 一张表决定代码放哪里

| 需求 | 推荐位置 | 不推荐位置 | 判断原因 |
| --- | --- | --- | --- |
| 单个页面的字段、按钮、弹窗 | 对应 `views` 页面或局部 `components` | 全局 Store、全局组件 | 生命周期只属于当前页面 |
| 多页面稳定复用的 UI 协议 | `src/components/Kh*` | 复制到多个页面 | 输入输出已脱离具体业务 |
| 单页内可复用的响应式逻辑 | `composables` | Store | 需要复用但不需要全局持久 |
| 用户、权限、字典、连接 | `stores` | 页面局部变量 | 多处共享且有应用级生命周期 |
| 标准 CRUD | `useCrudApi(module)` | 页面拼 URL | 端点和参数完全遵循标准 |
| 领域专用接口 | `src/api/<domain>.js` | CRUD 工厂塞特殊分支 | 方法应表达业务动作 |
| Token、续期、传输错误 | `request.js` | 每个 API 或页面 | 所有请求一致的协议行为 |
| 页面业务规则 | 页面交互校验 + 后端最终校验 | 请求拦截器 | 不属于全局传输层 |

## 12. 标准页面和例外页面怎么选

### 优先使用 KhPage

- 单表或近似单表维护。
- 查询、分页、新增、编辑、查看和删除遵循标准端点。
- 差异可以通过字段配置、插槽、事件或少量自定义按钮表达。

### 组合 KhTable、KhForm 和 KhDialog

- 主从表需要专用请求体。
- 编辑过程包含多个步骤或多个数据源。
- 页面仍以表格与表单为主，但 KhPage 生命周期不匹配。

### 编写专用交互页面

- 看板、拖拽、实时设备或可视化场景。
- 操作流程比 CRUD 更重要。
- 页面布局和数据刷新机制与标准维护页明显不同。

选择例外路径时，应在页面开头或评审说明中写清楚“哪个标准假设不成立”，而不是只写“需求特殊”。

## 13. 典型请求链路

以物料列表查询为例：

1. 权限 Store 从后端权限树生成 `basedata/material` 动态路由。
2. Router 匹配 `src/views/basedata/material.vue`。
3. 页面向 KhPage 提供 `module="material"`、列配置和权限前缀。
4. KhPage 调用 `useCrudApi('material').pageList`。
5. `buildPageQuery` 把页码、排序、查询字段和表头筛选转换为后端结构。
6. `request.js` 注入 Token，管理 Loading，并发送 `/api/material/pagelist`。
7. 后端返回统一响应，request.js 处理会话和错误后把数据交回 KhTable。
8. KhTable 更新行数据、分页总数和加载状态。

排错时按相反顺序检查：浏览器请求结果、请求参数转换、KhPage load、页面配置、动态路由映射和权限数据。

## 14. 架构失控信号

- 页面直接 `axios.create()`，绕过统一 Token 和错误处理。
- 同一种 CRUD 在不同页面使用不同分页和成功判断。
- 为一个页面向 KhPage 增加大量业务命名属性。
- 通用组件直接导入某个 `views` 页面或领域 API。
- 所有页面状态都进入 Pinia，退出页面后仍残留。
- 路由、菜单和按钮权限使用三套互不对应的编码。
- 前端把后端业务失败吞掉并显示固定“操作成功”。
- `config.js` 中写入秘密信息，或把运行时地址重新固化到业务代码。

## 15. 前端架构评审清单

- 页面属于哪个业务目录，局部组件是否跟随页面放置？
- 是标准 CRUD 还是明确的例外页面？
- 是否复用了唯一请求实例和统一响应判断？
- `module`、后端路由、菜单 component、页面路径和权限前缀是否对应？
- 状态是否真的需要跨页面；退出或切换用户时如何清理？
- 通用组件新增能力是否至少服务多个稳定场景？
- 前端校验失败和后端业务失败是否都能正确展示？
- PC、PDA、窄屏和刷新恢复是否符合页面使用场景？
