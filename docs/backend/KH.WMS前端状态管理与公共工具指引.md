## 2. 什么时候写 Store

适合 Store：

- 登录态。
- 用户信息。
- 权限和动态菜单。
- 字典缓存。
- WebSocket 连接状态。
- 多页面共享的全局状态。

不适合 Store：

- 只在一个弹窗里使用的临时表单状态。
- 某个页面独有的搜索条件。
- 可通过 props 向下传递的局部数据。

判断标准：

| 问题 | 如果答案是“是” |
| --- | --- |
| 刷新页面后是否需要恢复 | 倾向 Store + 持久化 |
| 多个页面是否同时使用 | 倾向 Store |
| 是否影响路由、权限、登录态 | 必须 Store |
| 是否只是弹窗打开期间使用 | 页面局部状态 |
| 是否能从接口重新获取 | 不一定要 Store，先考虑缓存成本 |

## 3. user Store

职责：

- 保存 `token`。
- 保存 `refreshToken`。
- 保存 `userInfo`。
- 登录、登出、获取当前用户信息。
- 同步 localStorage。

注意：

- 登出时即使 API 失败也要清除本地状态。
- 页面刷新后由路由守卫重新获取用户信息。
- Token 更新时要同步 Store 和 localStorage。

培训时要强调：`user` Store 是登录态的唯一可信前端来源。页面不要各自从 localStorage 解析用户对象后自行判断，否则刷新、退出、Token 更新时容易出现状态不一致。

## 4. permission Store

职责：

- 保存原始权限树。
- 提取路由权限。
- 提取按钮权限。
- 构建动态菜单。
- 构建动态路由。
- 提供 `hasRoutePermission` 和 `hasButtonPermission`。

注意：

- 权限未加载时默认放行，避免刷新初始阶段误拦截。
- `*` 表示拥有全部权限。
- 目录必须有子菜单，菜单项必须有路径。

权限 Store 的典型生命周期：

```text
登录成功
  -> 保存 token
  -> 获取用户信息
  -> 获取权限树
  -> 生成菜单和动态路由
  -> 页面访问时校验路由/按钮权限
刷新页面
  -> 根据 token 恢复用户信息
  -> 重新拉取权限树
  -> 重新注册动态路由
```

## 5. dict Store

职责：

- 按 `dictType` 缓存字典。
- 对同一字典请求去重。
- 支持刷新指定字典。
- 支持登出时清空缓存。

使用建议：

- 页面下拉优先走字典。
- 字典管理页修改字典后调用刷新。
- 不要在页面写死通用状态文案。

字典适合承载：

- 启停状态。
- 单据状态展示文案。
- 业务类型下拉项。
- 表格状态标签样式。

字典不适合承载：

- 复杂流程规则。
- 权限判断。
- 库存扣减逻辑。
- 需要事务保障的业务判断。

## 6. WebSocket

WebSocket 状态分两层：

- `src/utils/websocket.js`：连接、发送、事件处理等工具能力。
- `src/stores/websocket.js`：连接状态和跨页面共享数据。

开发约定：

- 工具层不放页面业务状态。
- Store 不直接写复杂底层连接细节。
- 页面通过 Store 或封装方法使用 WebSocket。

## 7. 扩展字段

扩展字段相关：

- 后端 `/form-config` 返回动态字段。
- 前端 `useExtFields.js` 负责消费和转换。

注意：

- 客户定制字段优先考虑扩展字段。
- 页面表单应能处理没有扩展字段的情况。
- 扩展字段失败不应导致整个页面无法打开，除非该字段是强业务必需。

扩展字段页面处理顺序：

1. 页面加载基础配置。
2. 调用 `/form-config` 获取扩展字段。
3. 将扩展字段合并进表单配置。
4. 提交时把扩展字段值放到约定字段结构。
5. 详情页按同一配置解析展示。

## 8. 公共工具边界

| 工具 | 使用边界 |
| --- | --- |
| `request.js` | 所有 HTTP 请求 |
| `crud.js` | 标准 CRUD 和分页查询 |
| `dict-resolve.js` | 字典值转标签/样式 |
| `mockData.js` | 演示或临时模拟数据 |
| `row-style-presets.js` | 表格行样式 |
| `useExtFields.js` | 扩展字段表单 |

工具函数设计原则：

| 原则 | 说明 |
| --- | --- |
| 输入输出清晰 | 不依赖页面隐式状态 |
| 不吞异常 | 业务异常交给调用方或统一拦截器 |
| 不写死业务模块 | 通用工具不要只服务一个模块 |
| 不替代 Service | 复杂业务流程应由后端保证一致性 |
| 可单独理解 | 成员能不打开具体页面也知道工具用途 |

## 9. 培训案例：状态应该放在哪里

以“入库单列表 + 详情弹窗 + 字典状态 + 当前用户”为例：

| 状态 | 推荐位置 | 原因 |
| --- | --- | --- |
| 当前登录用户 | `user` Store | 全局共享，刷新后恢复 |
| 当前用户按钮权限 | `permission` Store | 多页面共享 |
| 入库单状态字典 | `dict` Store | 多页面复用，可缓存 |
| 入库单搜索条件 | 当前页面 | 只服务列表页 |
| 当前打开的详情弹窗 ID | 当前页面 | 局部交互状态 |
| WebSocket 任务提醒 | `websocket` Store | 跨页面提示 |

培训练习：给成员一个页面需求，让他们先画出状态清单，再决定哪些放 Store，哪些留在页面。这样能避免 Store 变成“全局垃圾桶”。

## 10. 常见错误

| 错误 | 后果 | 正确做法 |
| --- | --- | --- |
| 页面局部状态放 Store | Store 膨胀 | 页面内维护 |
| 字典每次都请求 | 接口浪费 | 使用 dict Store 缓存 |
| 工具函数混入业务流程 | 难复用 | 业务流程放页面/业务模块 |
| 登出不清 Store | 数据串号 | 清 token、权限、字典 |
| 扩展字段异常未兜底 | 页面打不开 | 做空数组/默认配置兜底 |
