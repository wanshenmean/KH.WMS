# 02 模块边界与分层职责

## 这个概念解决什么问题

模块边界解决的是“代码应该放在哪里”和“模块之间怎么协作”的问题。

KH.WMS 后端按职责拆分为：

- `KH.WMS.Server`：应用启动、HTTP 管道、Swagger、后台服务。
- `KH.WMS.Core`：框架底座，包括 DI、AOP、数据库、事务、统一响应、鉴权、缓存、日志、CRUD 基类。
- `KH.WMS.Entities`：业务实体、常量、枚举。
- `KH.WMS.Contracts`：跨模块公共契约接口和请求模型。
- `KH.WMS.Modules.*`：具体业务模块，例如入库、出库、库存、任务、仓库、基础资料、系统。
- `KH.WMS.Config`：配置模块和配置库实体，提供配置解析、编码规则、单据状态、扩展字段等底座能力。
- `KH.WMS.Algorithms`：策略算法和策略链执行底座。

分层职责清楚后，业务代码会更容易维护：Controller 不写业务流程，Service 不直接依赖别的模块内部实现，跨模块协作通过 Contract 完成。

## 什么时候需要看

- 新增业务模块、Controller、Service、DTO、Entity。
- 不确定某段逻辑应该放 Controller、Service 还是 Contract。
- 一个模块需要调用另一个模块能力，例如入库创建任务、出库扣减库存。
- 想复用配置、策略、编码、状态能力，但不确定能不能直接引用实现类。
- 排查模块间循环引用、代码越界依赖、Controller 越写越重的问题。

## 业务开发应该怎么用

### Controller 的职责

Controller 只处理 HTTP 层：

- 定义路由和 Action。
- 接收请求 DTO 或实体。
- 调用 Service。
- 返回 `ApiResponse` 或 `ServiceResult` 转换后的结果。

不要在 Controller 里写复杂业务流程、事务、仓储查询、跨模块调用。普通 CRUD Controller 应优先继承 `CrudController<TEntity>` 或 `ExtDataCrudController<TEntity>`。

### Service 的职责

Service 是业务流程的主入口：

- 实现本模块业务规则。
- 调用本模块仓储。
- 调用跨模块 Contract。
- 控制复杂流程的事务边界。
- 重写 CRUD 钩子实现定制逻辑。

Service 一般放在模块 `Services/` 下，并实现模块内 `Interfaces/` 下的接口。

### Entity 的职责

Entity 表示数据库持久化结构：

- 字段映射。
- 主从表导航关系。
- 简单领域方法，例如实体状态变更方法。

不要让 Entity 依赖 Service、Repository、HttpContext 或其他基础设施。

### DTO 的职责

DTO 是接口出入参或页面需要的数据形状：

- 创建、更新请求。
- 查询返回。
- 聚合页面展示。
- WCS、第三方接口请求模型。

不要为了方便直接把复杂页面字段全部塞进 Entity。Entity 是持久化模型，DTO 是接口模型。

### Contract 的职责

Contract 是跨模块调用边界：

- 接口放 `KH.WMS.Contracts` 或 `KH.WMS.Config.Abstractions`。
- 实现放被调用模块的 `Contracts/` 目录。
- 调用方只注入接口，不引用对方模块 Service。

例如入库模块要创建上架任务，应注入 `ITaskContract`，而不是引用 `TaskHeaderService`。

## 底层逻辑和实现

### 模块项目怎么被发现

Controller 扫描依赖程序集名约定：

- `.Modules.` 命中的程序集会被加入 MVC `ApplicationPart`。
- `KH.WMS.Config` 被启动入口单独加入。

Service 扫描依赖注册特性：

- `[RegisteredService]` 用于接口注册。
- `[SelfRegisteredService]` 用于自注册。
- `ServiceRegistrar` 会把注册的服务挂上 AOP 拦截器，除非设置 `WithoutInterceptor=true`。

### Config 模块为什么特殊

`KH.WMS.Config` 不是普通业务模块模板，它更像配置底座：

- 配置实体使用 `[ConfigDb]` 路由到 SQLite 配置库。
- 对外提供配置解析、扩展字段、编码规则、单据状态校验等 Contract。
- 业务模块应该调用 Config 的抽象接口，而不是复制 Config 模块结构。

### Algorithms 模块为什么单独放

策略算法会被多个业务模块调用，例如入库上架、货位分配、出库库存分配。它的重点不是某个业务单据的 CRUD，而是“按配置选择和执行策略”。

业务模块只需要通过 `IPolicyEngine` 调用策略，不应该直接 new 某个策略实现。

### `ModuleBase` / `ModuleLoader` 怎么看

`KH.WMS.Core.Modularity` 里有 `ModuleBase`、`ModuleLoader`、`ModuleExtensions` 等预留框架能力，用于模块生命周期、依赖排序和注册扩展。

当前业务主链路并没有在 `Program.cs` 中调用 `RegisterModules()` 或 `InitializeModulesAsync()`。也就是说，目前业务模块主要靠程序集扫描、注册特性和 MVC `ApplicationPart` 接入。业务开发了解即可，不需要为每个模块补 `ModuleBase`。

## 落代码决策表

| 你要写的东西 | 应该放哪里 | 不应该放哪里 | 判断标准 |
| --- | --- | --- | --- |
| 一个 HTTP 接口 | 模块 `Controllers/` | `Core`、其他模块 Controller | 只负责路由、入参、调用 Service |
| 页面新增/编辑/删除流程 | 模块 `Services/` | Controller | 需要仓储、事务、状态、跨模块调用 |
| 数据库表结构 | `KH.WMS.Entities` 或 Config 实体目录 | DTO、Controller | 需要持久化和 ORM 映射 |
| 页面请求/响应模型 | 模块 `DTOs/` | Entity | 页面字段和数据库字段不完全一致 |
| 跨模块能力接口 | `KH.WMS.Contracts` 或 `Config.Abstractions` | 被调用模块 `Interfaces/` | 其他模块要引用 |
| 跨模块能力实现 | 被调用模块 `Contracts/` | 调用方模块 | 数据和规则属于哪个模块，谁实现 |
| 配置解析、状态、扩展字段 | `KH.WMS.Config` | 业务模块复制一份 | 属于配置底座 |
| 策略选择和算法 | `KH.WMS.Algorithms` | 业务 Service 硬编码 | 决策可配置、可替换 |
| 框架横切能力 | `KH.WMS.Core` | 业务模块 | DI、AOP、事务、响应、鉴权、缓存 |

## 引用方向

推荐依赖方向：

```text
Server
  -> Core
  -> Entities
  -> Contracts
  -> Config
  -> Algorithms
  -> Modules.*

Modules.*
  -> Core
  -> Entities
  -> Contracts
  -> Config.Abstractions
  -> Algorithms.Interfaces
```

模块之间不要互相直接引用实现项目。比如 `InboundModule` 不应该直接引用 `TaskModule.Services.TaskHeaderService`，而是依赖 `KH.WMS.Contracts.Tasks.ITaskContract`。

如果你发现为了调用一个方法必须给两个模块互相加项目引用，通常说明边界错了，需要抽 Contract。

## 业务流程拆分示例

以“入库组盘后申请上架”为例，代码应该这样分层：

1. Controller 接收 `headerIds`，调用 `IInboundContainerBindService.RequestPutAwayAsync`。
2. 入库 Service 开启事务，加载组盘头和明细，校验组盘状态。
3. 入库 Service 构造 `PolicyContext`，调用 `IPolicyEngine` 执行上架策略。
4. 入库 Service 构造 `PutawayTaskRequest`，调用 `ITaskContract.CreatePutawayTaskAsync`。
5. Task 模块 Contract 生成任务号，写任务头和任务行。
6. 入库 Service 调用容器 Contract 更新容器状态。
7. 入库 Service 提交事务，返回 `ServiceResult`。

这条链路里：

- Controller 没有业务细节。
- 入库模块负责流程编排。
- Task 模块只负责任务自己的数据。
- 策略模块只负责决策。
- Contract 是模块间边界。

## 模块边界坏掉时的信号

- Controller 构造函数里注入了 5 个以上 Repository。
- 一个 Service 同时直接引用多个其他模块的具体 Service。
- 为了一个跨模块调用新增了项目循环引用。
- DTO 被多个模块直接复用，且字段开始互相污染。
- Config 表被业务模块当普通业务表写入流水数据。
- 策略判断写成大量 `if warehouseId == ...`，而不是配置策略链。

出现这些信号时，先停下来重新划分边界，通常比继续堆代码更省时间。

## 最小示例

一个普通业务模块通常是这个结构：

```text
KH.WMS.Modules.SampleModule/
  Controllers/
    SampleController.cs
  Interfaces/
    ISampleService.cs
  Services/
    SampleService.cs
  DTOs/
    SampleDto.cs
  Contracts/
    SampleContract.cs
```

Service 写法：

```csharp
[RegisteredService(ServiceType = typeof(ISampleService))]
public class SampleService(
    IRepository<SampleEntity, long> repository,
    IUnitOfWork unitOfWork,
    IDetailSaveService detailSaveService)
    : CrudService<SampleEntity>(repository, unitOfWork, detailSaveService),
      ISampleService
{
    protected override Task BeforeCreateAsync(SampleEntity entity)
    {
        // 写本模块新增前校验
        return Task.CompletedTask;
    }
}
```

Controller 写法：

```csharp
[Route("api/sample")]
public class SampleController(ISampleService service)
    : CrudController<SampleEntity>(service)
{
}
```

## 常见误区与排查清单

- Controller 里写事务和仓储查询：优先移到 Service。
- Service 直接注入另一个模块的具体 Service：改成注入 Contract。
- Entity 里注入服务：Entity 只保留数据和少量领域方法。
- DTO 和 Entity 混用：创建、编辑、页面聚合优先使用 DTO。
- 复制 Config 模块做业务模块：Config 是底座模块，不是普通业务模板。
- 新模块命名不含 `.Modules.`：Controller 可能不会被 MVC 扫描到。
- 以为实现 `ModuleBase` 才能接入模块：当前主链路不要求。
