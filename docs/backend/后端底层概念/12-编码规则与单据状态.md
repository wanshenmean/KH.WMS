# 12 编码规则与单据状态

## 这个概念解决什么问题

编码规则解决的是“单号、任务号、物料编码等业务编码如何按配置生成”的问题。单据状态解决的是“业务流程中的状态能否流转、能否编辑、能否删除”的问题。

这两个能力都属于业务流程的基础规则：

- 编码不能散落在每个 Service 里手拼。
- 状态不能靠每个模块硬编码 if/else。

KH.WMS 通过配置库维护规则，再由服务统一读取和执行。

## 什么时候需要看

- 新增单据号、任务号、物料编码、容器编码等生成逻辑。
- 生成单号报“未找到有效编码规则”。
- 单号重复、日期序列没有重置、编码格式不符合预期。
- 新增单据状态或调整流转顺序。
- 业务流程提交、审核、完成、取消时需要校验状态。
- 编辑或删除按钮需要按状态控制。

## 业务开发应该怎么用

### 生成编码

业务 Service 注入 `ICodeGeneratorService`：

```csharp
var taskNo = await _codeGenerator.GenerateAsync(BizConstants.CodeRuleTypes.TASK_NO);
```

不要在业务模块里手动查询 `CfgCodeRule`、拼日期、维护序号。

`GenerateAsync` 当前只有一个参数 `ruleType`。这意味着调用方只能按“规则类型”取号，不能指定某一条具体 `RuleCode`。如果一个 `RuleType` 下配置了多条启用且默认的规则，代码只会 `LIMIT 1` 取到其中一条，命中哪条取决于数据库返回顺序，不应该依赖这种行为。

开发时要保证：

- 同一个 `RuleType` 在当前生效日期内只有一条 `IsActive=1` 且 `IsDefault=1` 的规则。
- 需要切换编码格式时，优先调整旧规则失效日期、新规则生效日期，而不是让两条默认规则同时生效。
- 业务代码里传常量，例如 `BizConstants.CodeRuleTypes.INBOUND_DOC`、`TASK_NO`，不要散落字符串。

### 校验状态流转

业务 Service 注入 `IDocumentStatusValidatorContract`：

```csharp
await _statusValidator.ValidateTransitionAsync(docTypeCode, order.OrderStatus, nextStatus);
order.SetStatus(nextStatus);
```

如果只是判断按钮可用或预检查，可以用：

```csharp
var canSubmit = await _statusValidator.CanTransitionAsync(docTypeCode, currentStatus, targetStatus);
```

### 校验能否编辑或删除

```csharp
await _statusValidator.ValidateAllowEditAsync(docTypeCode, currentStatus);
await _statusValidator.ValidateAllowDeleteAsync(docTypeCode, currentStatus);
```

这些方法不通过时会抛异常，由全局异常处理统一返回。

## 底层逻辑和实现

### 编码规则数据来源

`CodeGeneratorService` 读取配置库：

- `cfg_code_rule`：编码规则。
- `cfg_code_sequence`：序列当前值和周期。

实体对应：

- `CfgCodeRule`
- `CfgCodeSequence`

这些实体标记 `[ConfigDb]`，存放在配置库。

### 编码生成流程

`GenerateAsync(ruleType)` 大致流程：

1. 进入进程内串行锁 `SemaphoreSlim`。
2. 查询有效的默认规则：
   - `RuleType` 匹配。
   - `IsActive = 1`。
   - `IsDefault = 1`。
   - 当前日期在生效期内。
3. 根据 `SequenceType` 生成序列 key：
   - `DAILY`：`ruleType_yyyyMMdd`
   - `MONTHLY`：`ruleType_yyyyMM`
   - `YEARLY`：`ruleType_yyyy`
   - 其他：`ruleType_NONE`
4. 查询或创建 `cfg_code_sequence`。
5. 周期过期时重置序列。
6. 当前值加 1。
7. 回写序列表。
8. 按前缀、日期格式、分隔符、序列长度拼出编码。
9. 释放锁。

这里有几个实现细节要知道：

- `DateFormat` 走 .NET 日期格式，例如 `yyyyMMdd`、`yyyyMM`，不是数据库日期格式。
- `SequenceLength` 只负责左侧补零，不负责截断。序列值超过长度后会原样变长，例如长度 4 时第 10000 个号是 `10000`。
- `CacheSize` 和 `Example` 是配置表字段，但当前 `CodeGeneratorService` 生成逻辑没有使用它们。不要以为配置了 `CacheSize` 就会预取号段。
- `RuleCode` 是规则自身编码，当前生成时不按 `RuleCode` 查询。
- `Prefix`、`DateFormat` 为空时会跳过对应片段，最终至少会拼上序列号。

### 并发边界

当前取号使用进程内 `SemaphoreSlim` 串行化，适合当前单实例 Windows 服务部署。

如果未来多实例部署，进程内锁不能跨实例保证唯一性，需要改成数据库层原子自增、唯一约束或数据库序列方案。这是架构边界，不是业务 Service 应该自己补的逻辑。

配置库表 `cfg_code_sequence` 实体上已经定义了 `(RuleId, SequenceKey)` 唯一索引。它能防止同一规则同一周期出现两条序列记录，但不能替代跨进程的“读取当前值、加一、写回”原子性。多实例下仍然需要调整取号实现。

### 单据状态数据来源

`DocumentStatusValidatorContract` 查询：

- `CfgDocumentType`
- `CfgDocumentStatus`

状态配置中重要字段：

- `StatusCode`：状态编码。
- `StatusName`：状态名称。
- `NextStatuses`：允许流转到的目标状态 JSON 数组。
- `AllowEdit`：当前状态是否允许编辑。
- `AllowDelete`：当前状态是否允许删除。
- `IsInitial`：是否初始状态。
- `IsActive`：是否启用。

### 状态校验流程

`ValidateTransitionAsync(docTypeCode, fromStatus, toStatus)`：

1. 如果 from 和 to 相同，直接通过。
2. 根据单据类型编码找单据类型。
3. 查当前状态配置。
4. 解析 `NextStatuses`。
5. 如果目标状态不在列表里，抛 `InvalidOperationException`。

`CanTransitionAsync` 是不抛异常版本，内部 catch 后返回 bool。

状态配置会缓存 30 分钟：

- 单据类型缓存 key：`Data:DocType:Code:{docTypeCode}`。
- 状态配置缓存 key：`Data:DocStatus:{docTypeCode}:{status}`。

如果刚在配置表里修改了状态流转，但业务接口仍按旧规则判断，先考虑缓存未过期。开发联调时可以重启服务、清缓存，或等待缓存过期后再验证。

`NextStatuses` 是 JSON 数组字符串。解析失败时当前实现不会抛异常，而是按空列表处理：

```json
["RECEIVING", "CANCELLED"]
```

如果误写成 `RECEIVING,CANCELLED` 或对象格式，效果通常就是“任何目标状态都不允许流转”。排查状态流转失败时，一定要先检查 `NextStatuses` 是否为合法 JSON 数组。

### 状态相关方法怎么选

| 方法 | 是否抛异常 | 适合场景 |
| --- | --- | --- |
| `ValidateTransitionAsync` | 不允许时抛异常 | 提交、确认、收货、上架完成等真正要推进状态的写流程 |
| `CanTransitionAsync` | 不抛异常，返回 `bool` | 按钮可用性、预检查、页面提示 |
| `GetAllowedTransitionsAsync` | 不抛异常，返回目标状态列表 | 需要让调用方从允许状态里选择下一状态 |
| `GetStatusConfigAsync` | 找不到返回 `null` | 需要读取 `AllowEdit`、`AllowDelete`、颜色、状态名等配置 |
| `ValidateAllowEditAsync` | 不允许时抛异常 | 更新接口进入前校验 |
| `ValidateAllowDeleteAsync` | 不允许时抛异常 | 删除接口进入前校验 |
| `GetInitialStatusAsync` | 找不到返回 `null` | 创建单据时决定初始状态 |

写流程里优先用 `Validate...` 系列，让异常交给全局异常处理和事务逻辑；展示或预检查用 `Can...` / `Get...` 系列。

## 编码规则字段如何影响结果

生成编码时最终拼装由这些字段决定：

| 字段 | 作用 | 示例 |
| --- | --- | --- |
| `RuleType` | 调用时传入的规则类型 | `TASK_NO` |
| `Prefix` | 编码前缀 | `TASK` |
| `DateFormat` | 日期部分格式 | `yyyyMMdd` |
| `Separator` | 前缀、日期、序列之间的分隔符 | `-` |
| `SequenceLength` | 序列补零长度 | `5` |
| `SequenceType` | 序列周期 | `DAILY` |

例如：

```text
Prefix = TASK
DateFormat = yyyyMMdd
Separator = -
SequenceLength = 5
SequenceType = DAILY
CurrentValue = 12
```

生成结果类似：

```text
TASK-20260707-00013
```

如果业务觉得编码格式不对，先看配置，不要先改代码。

## 序列表的周期行为

`cfg_code_sequence.SequenceKey` 决定一条序列记录的粒度：

| SequenceType | SequenceKey 示例 | 重置时机 |
| --- | --- | --- |
| `DAILY` | `TASK_NO_20260707` | 每天 |
| `MONTHLY` | `TASK_NO_202607` | 每月 |
| `YEARLY` | `TASK_NO_2026` | 每年 |
| 其他 | `TASK_NO_NONE` | 不按日期周期重置 |

周期过期时，服务会把 `CurrentValue` 重置为 0，并更新周期开始和结束。下一次生成再自增。

## 单据状态配置示例

假设入库单状态：

```text
DRAFT -> RECEIVED -> BOUND -> COMPLETED
```

那么 `CfgDocumentStatus` 可以这样配置：

| StatusCode | NextStatuses | AllowEdit | AllowDelete |
| --- | --- | --- | --- |
| `DRAFT` | `["RECEIVED"]` | 1 | 1 |
| `RECEIVED` | `["BOUND"]` | 0 | 0 |
| `BOUND` | `["COMPLETED"]` | 0 | 0 |
| `COMPLETED` | `[]` | 0 | 0 |

业务提交时不应该写：

```csharp
if (order.OrderStatus != "RECEIVED") ...
```

更好的方式是：

```csharp
await _statusValidator.ValidateTransitionAsync(docTypeCode, order.OrderStatus, targetStatus);
```

这样调整流转规则时改配置即可。

## 状态校验和实体方法的配合

推荐顺序：

1. Contract 校验配置允许流转。
2. 实体方法执行状态变更。
3. Repository 保存。

示例：

```csharp
await _statusValidator.ValidateTransitionAsync(docTypeCode, order.OrderStatus, boundStatus);
order.MarkAsBound(boundStatus);
await _orderRepository.UpdateAsync(order);
```

状态校验负责“能不能从 A 到 B”，实体方法负责“从 A 到 B 时实体字段怎么变”。两者不要混成一坨。

## 排查单号重复

按顺序看：

1. 是否单实例部署。当前锁只在进程内有效。
2. 配置库中 `cfg_code_sequence` 是否有重复 `RuleId + SequenceKey`。
3. 是否有业务绕过 `ICodeGeneratorService` 自己生成。
4. 是否数据库缺少唯一约束兜底。
5. 是否多套服务共用同一个配置库并发取号。

## 最小示例

### 创建任务时生成任务号

```csharp
var taskNo = await codeGenerator.GenerateAsync(BizConstants.CodeRuleTypes.TASK_NO);

var taskHeader = new TaskHeader
{
    TaskNo = taskNo,
    TaskStatus = BizConstants.TaskStatus.PENDING
};
```

### 入库组盘完成后推进订单状态

```csharp
var docTypeCode = OrderTypeMappings.ToDocTypeCode(order.OrderType);
var nextStatus = BizConstants.InboundOrderStatus.BOUND;

await _statusValidator.ValidateTransitionAsync(
    docTypeCode,
    order.OrderStatus,
    nextStatus);

order.MarkAsBound(nextStatus);
await _orderRepository.UpdateAsync(order);
```

### 获取初始状态

```csharp
var initialStatus = await _statusValidator.GetInitialStatusAsync(docTypeCode)
    ?? BizConstants.InboundOrderStatus.DRAFT;
```

## 常见误区与排查清单

### 编码生成失败

- `ruleType` 是否和配置表一致。
- 是否存在启用且默认的编码规则。
- 生效日期和失效日期是否覆盖当前日期。
- 配置库是否初始化了 `cfg_code_rule` 表。

### 编码格式不对

- 检查 `Prefix`、`DateFormat`、`Separator`、`SequenceLength`。
- 检查 `SequenceType` 是 DAILY、MONTHLY、YEARLY 还是 NONE。
- 检查是否命中了错误的默认规则。

### 状态流转失败

- 单据类型编码是否正确。
- 当前状态配置是否启用。
- `NextStatuses` 是否包含目标状态。
- 是否应该先取允许目标状态列表，再选择业务状态。

### 多实例部署风险

- 当前取号锁是进程内锁。
- 多服务实例同时生成单号时需要数据库层唯一性方案。
- 不要在业务 Service 层临时加 `lock` 试图解决跨实例问题。
