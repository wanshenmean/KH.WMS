# 01 启动入口与程序集扫描

## 这个概念解决什么问题

启动入口负责把后端运行起来，并把散落在不同项目里的 Controller、Service、数据库、缓存、鉴权、Swagger、中间件和后台服务接到同一个 ASP.NET Core 应用里。

在 KH.WMS 里，业务模块不是靠手写一堆 `services.AddScoped<...>()` 接入的，而是主要依赖两条扫描链路：

- MVC Controller 扫描：让模块里的 Controller 能被路由和 Swagger 发现。
- Autofac 服务扫描：让带 `[RegisteredService]`、`[SelfRegisteredService]` 的服务自动注册进 DI 容器，并挂上 AOP 代理。

这两条链路互相独立。Service 能注入，不代表 Controller 一定能被 Swagger 看到；Swagger 能看到 Controller，也不代表 Controller 依赖的 Service 已经注册成功。

## 什么时候需要看

- 新增了模块 Controller，但是 Swagger 里没有接口。
- 新增了 Service 或 Contract，但是运行时报 `Unable to resolve service`。
- 新建了模块项目，接口在本地能编译，运行时却没有加载到。
- 修改了启动、Swagger、模块命名、项目引用、发布方式。
- 排查 `KH.WMS.*.dll` 缺失、F6 编译后才生效、模块 DLL 没复制到输出目录的问题。

## 业务开发应该怎么用

### 新增业务模块时

模块项目建议遵守当前约定：

- 项目名包含 `.Modules.`，例如 `KH.WMS.Modules.InventoryModule`。
- Controller 放在模块项目的 `Controllers/` 下。
- Service 放在模块项目的 `Services/` 下，并加 `[RegisteredService]`。
- 对外跨模块能力放在模块项目 `Contracts/` 下实现，接口放到 `KH.WMS.Contracts` 或 `KH.WMS.Config.Abstractions`。

只要模块 DLL 被引用并复制到运行目录，启动时会被扫描到。

### 新增 Controller 时

Controller 被发现依赖 MVC 的 `ApplicationPart`：

```csharp
.ConfigureApplicationPartManager(apm =>
{
    var moduleAssemblies = AssemblyService.GetReferencedAssemblies()
        .Where(a => a.GetName().Name?.Contains(".Modules.") == true
            || a.GetName().Name == "KH.WMS.Config");

    foreach (var assembly in moduleAssemblies)
    {
        apm.ApplicationParts.Add(new AssemblyPart(assembly));
    }
});
```

所以普通业务模块要让程序集名命中 `.Modules.`。`KH.WMS.Config` 是配置模块特例，虽然名称不含 `.Modules.`，但启动入口显式加入。

### 新增 Service 时

Service 被发现依赖 Autofac 模块：

```csharp
builder.Host.UseServiceProviderFactory(new AutofacServiceProviderFactory())
    .ConfigureContainer<ContainerBuilder>(builder =>
    {
        builder.RegisterModule(new ServiceExtensions());
        builder.RegisterModule(new StrategyAutofacModule());
    });
```

`ServiceExtensions` 会通过 `AssemblyService.GetReferencedAssemblies()` 找到项目内程序集，再交给 `ServiceRegistrar` 扫描注册特性。

## 底层逻辑和实现

### 启动主线

核心启动过程在 `Program.cs`：

1. 创建 `WebApplicationBuilder`。
2. 切换 DI 容器为 Autofac。
3. 注册 `ServiceExtensions`，扫描带注册特性的服务。
4. 注册 `StrategyAutofacModule`，扫描并注册策略实现。
5. 调用 `AddInfrastructure` 注册数据库、缓存、认证、日志、监控、Swagger、CORS、限流配置、HTTP Client。
6. 注册 MVC Controller，并手动加入模块 Controller 程序集。
7. 启动后预热配置缓存和系统参数缓存。
8. 启用请求体缓冲，便于异常日志和 `ExtDataCrudController` 重读请求体。
9. 调用 `UseCustomMiddleware` 挂载异常处理、License、CORS、请求日志、路由、MiniProfiler、认证、授权和端点映射。
10. 启用 Swagger、静态文件和 SPA fallback。

### `AddInfrastructure` 做什么

`AddInfrastructure` 是基础设施统一入口，位置在 `KH.WMS.Core.Setup.ServiceCollectionSetup`。它注册的是框架级能力：

- SqlSugar 数据库。
- 缓存服务。
- JWT 认证。
- 日志。
- MiniProfiler。
- Swagger。
- CORS。
- 限流配置。
- HTTP Client。
- MVC Filter。

注意：这里不是业务 Service 的主要注册点。业务 Service 主要靠 Autofac `ServiceRegistrar` 自动注册。

### `AssemblyService.GetReferencedAssemblies()` 做什么

它从运行时依赖上下文里找项目内程序集：

- `Type == "project"` 的运行时库。
- 名称以 `KH.WMS.` 开头的库。

然后从 `AppContext.BaseDirectory` 加载对应 DLL。代码里有明确提示：项目解耦后，如果 DLL 没在输出目录，需要先 F6 编译，再 F5 运行。

这也是为什么“代码写好了但运行时找不到模块”经常不是代码问题，而是模块 DLL 没输出或没被引用。

## 真实执行链路

### 一个新接口从代码到 Swagger 的链路

假设你新增了 `KH.WMS.Modules.SampleModule.Controllers.SampleController`：

1. `SampleModule` 项目被 `KH.WMS.Server` 或其他项目引用。
2. 编译后输出目录里出现 `KH.WMS.Modules.SampleModule.dll`。
3. `AssemblyService.GetReferencedAssemblies()` 从运行时依赖里找到这个 DLL 并 `Assembly.LoadFrom`。
4. `Program.cs` 的 `ConfigureApplicationPartManager` 过滤出程序集名包含 `.Modules.` 的程序集。
5. MVC 把该程序集加入 `ApplicationParts`。
6. MVC Controller 发现机制扫描 `SampleController`。
7. `app.UseEndpoints(endpoints => endpoints.MapControllers())` 映射 Controller 路由。
8. `UseSwaggerDocumentation` 读取 API Explorer 信息，Swagger 页面显示接口。

如果第 2 步失败，Swagger 看不到接口；如果第 5 步失败，Swagger 也看不到接口；如果第 7 步被改坏，运行时路由不可达。

### 一个新 Service 从代码到可注入的链路

假设你新增了 `SampleService : ISampleService`：

1. `SampleService` 加 `[RegisteredService(ServiceType = typeof(ISampleService))]`。
2. 模块 DLL 被 `AssemblyService.GetReferencedAssemblies()` 加载。
3. Autofac 执行 `ServiceExtensions.Load`。
4. `ServiceRegistrar.Register` 扫描程序集类型。
5. `RegisterServicesWithLifetime` 找到注册特性。
6. 根据 `ServiceType` 决定服务暴露接口。
7. 根据 `Lifetime` 注册生命周期。
8. 如果没有 `WithoutInterceptor=true`，挂上接口拦截器。
9. Controller 或其他 Service 构造函数注入 `ISampleService` 时解析成功。

Controller 扫描和 Service 注入分别走两套链路，所以排查时要分开看。

### 启动时预热失败为什么不一定导致启动失败

`Program.cs` 里配置预热和系统参数预热被 `try { await ... } catch { }` 包住：

```csharp
try { await configService.WarmUpAsync(); } catch { }
try { await paramService.WarmUpAsync(); } catch { }
```

这表示预热失败不会阻断服务启动，但会导致首次请求时再查数据库，或者某些运行期开关读不到缓存值。排查配置异常时，不要只看应用是否启动成功，还要看预热阶段有没有被吞掉的异常线索。

## 排查打法

### Swagger 没接口时按层排查

1. 看 DLL：输出目录是否有模块 DLL。
2. 看项目引用：Server 是否能把模块带入运行时依赖。
3. 看程序集名：是否命中 `.Modules.` 或 `KH.WMS.Config`。
4. 看 Controller：是否 public、非 abstract、路由和 `[ApiController]` 是否符合 MVC 发现规则。
5. 看端点映射：`UseCustomMiddleware` 里是否仍有 `endpoints.MapControllers()`。
6. 看 Swagger：是否只是 Swagger 文档过滤问题，而接口实际可访问。

### 服务注入失败时按层排查

1. 实现类是否加了注册特性。
2. 注册特性是否暴露了正确接口。
3. 实现类是否在已加载程序集里。
4. 构造函数依赖是否也都能解析。
5. 是否因为单例依赖 Scoped 服务导致构造异常。
6. 是否多个实现注册到同一个接口，调用方期望的是单个实例还是集合。

### 启动阶段建议看的日志或断点

- `AssemblyService.GetReferencedAssemblies()`：确认加载了哪些 DLL。
- `ServiceRegistrar.RegisterServicesWithLifetime()`：确认扫描到了哪些 `[RegisteredService]`。
- `Program.cs` 的 `ConfigureApplicationPartManager`：确认 Controller 程序集是否加入 `ApplicationParts`。
- `UseCustomMiddleware`：确认请求管道是否完整挂载。

## 常见误区与排查清单

### Swagger 看不到接口

- Controller 所在项目名是否包含 `.Modules.`，或是否是显式加入的 `KH.WMS.Config`。
- 模块项目是否被 Server 项目引用。
- 运行目录是否存在对应 `KH.WMS.Modules.XxxModule.dll`。
- Controller 是否继承 `ControllerBase` 或 CRUD 基类，是否有 `[ApiController]`、路由特性。
- 是否只新增了 Service，没有新增 Controller。Service 注册成功不会自动生成接口。

### Service 注入失败

- 实现类是否加了 `[RegisteredService]` 或 `[SelfRegisteredService]`。
- 接口是否在实现类上真的实现了。
- 有多个接口时是否通过 `ServiceType` 指定了要暴露的接口。
- 模块 DLL 是否被 `AssemblyService` 加载。
- 如果是泛型服务，确认注册的是开放泛型定义，例如 `CrudService<>`。

### 代码改了但运行结果没变

- 先确认当前运行的是哪个输出目录。
- 重新编译整个解决方案，尤其是模块项目。
- 清理旧 DLL 后再运行，避免运行目录残留旧版本。

### 不要这样做

- 不要在业务模块里手动复制 `Program.cs` 的基础设施注册。
- 不要为了 Swagger 可见性把 Controller 挪进 Server 项目。
- 不要绕开自动注册，在多个地方重复注册同一个 Service，除非你清楚注册覆盖顺序。
