# 11 可插拔校验器

## 这个概念解决什么问题

可插拔校验器解决的是“可配置的前置业务校验不要写死在主流程里”的问题。

例如入库组盘时，有些规则可能因客户或配置不同而变化：

- 是否允许空提交。
- 组盘数量是否超出订单行。
- 是否要求批次号。
- 是否要求效期。
- 是否允许混料。
- 是否允许混批。

这些规则被拆成独立 `IValidator`，业务方法通过 `[ConfigValidation]` 声明需要哪些校验器。拦截器在方法执行前运行校验链，失败时短路返回 `ServiceResult.Fail`。

## 什么时候需要看

- 要新增一个配置驱动的业务校验。
- `[ConfigValidation]` 加了但业务方法仍然执行。
- 校验失败没有返回预期错误。
- Validator 里需要读取全局配置。
- 不确定校验应该写在 `BeforeCreateAsync` 还是独立 Validator。
- 遇到 Validator 递归拦截或依赖注入问题。

## 业务开发应该怎么用

### 适合做成 Validator 的规则

- 规则可复用。
- 规则依赖配置开关。
- 规则是业务方法执行前的前置检查。
- 失败后不应该执行原业务方法。
- 规则之间有明确执行顺序。

### 不适合做成 Validator 的规则

- 只在某个 CRUD 新增前使用一次，写 `BeforeCreateAsync` 更简单。
- 规则需要修改实体状态或写数据库。
- 规则属于业务流程中间步骤，而不是前置检查。
- 规则需要返回复杂响应，不只是错误消息。

### 业务方法怎么声明校验

```csharp
[ConfigValidation(ValidatorCodes.BIND_DATA_NOT_EMPTY)]
[ConfigValidation(ValidatorCodes.BIND_QUANTITY)]
[ConfigValidation(ValidatorCodes.MIXED_MATERIAL)]
public async Task<ServiceResult> ContainerBindAsync(List<ContainerBindDto> binds)
{
    ...
}
```

多个特性按声明顺序执行。任一校验返回错误消息，原业务方法不会执行。

### Validator 怎么写

```csharp
[RegisteredService(WithoutInterceptor = true, ServiceType = typeof(IValidator))]
public class MixedMaterialValidator : IValidator
{
    public string Code => ValidatorCodes.MIXED_MATERIAL;

    public async Task<string?> ValidateAsync(object?[] args, Dictionary<string, object>? services = null)
    {
        ...
        return null; // 通过
    }
}
```

返回规则：

- `null` 表示通过。
- 非空字符串表示失败，字符串会作为 `ServiceResult.Fail(message)` 的消息。

## 底层逻辑和实现

### 三个核心类型

| 类型 | 作用 |
| --- | --- |
| `ConfigValidationAttribute` | 标在业务方法上，声明要执行哪个校验器 |
| `ValidatorCodes` | 校验器编码常量，避免字符串散落 |
| `IValidator` | 所有校验器统一接口 |

### 拦截器执行链

`ConfigValidationInterceptor` 执行逻辑：

1. 从目标方法读取 `[ConfigValidation]`。
2. 如果没有特性，直接执行原方法。
3. 检查返回类型。
4. 解析当前作用域内所有 `IValidator`。
5. 用 `IValidator.Code` 建字典。
6. 通过类型名动态解析 `IConfigResolverContract`，放入 `services["ConfigService"]`。
7. 按特性声明顺序执行 Validator。
8. 任一返回错误消息时，构造 `Task<ServiceResult>` 或 `Task<ServiceResult<T>>` 的失败结果并短路。
9. 全部通过后执行原业务方法。

### 返回类型限制

当前拦截器只对以下返回类型短路：

- `Task<ServiceResult>`
- `Task<ServiceResult<T>>`

如果方法返回 `Task<ApiResponse>` 或其他类型，拦截器会直接放行。

这就是为什么 ConfigValidation 更适合服务内部流程方法，而不是标准 CRUD Controller Action。

### 为什么 Validator 要 `WithoutInterceptor=true`

Validator 是被拦截器解析并执行的。如果 Validator 自己也挂 AOP，可能导致：

- 递归拦截。
- 额外日志噪声。
- 拦截器依赖链复杂化。
- 性能浪费。

所以 Validator 示例都应使用：

```csharp
[RegisteredService(WithoutInterceptor = true, ServiceType = typeof(IValidator))]
```

### 配置服务怎么传给 Validator

`ValidateAsync(object?[] args, Dictionary<string, object>? services = null)` 里的 `services` 是拦截器给 Validator 准备的“上下文服务字典”。它不是业务方法的参数，也不是前端传进来的数据。

它的设计目的很简单：`IValidator` 定义在 `KH.WMS.Core`，而某些校验器需要读取配置底座能力。如果直接让 Core 引用 Config，会把底层项目和配置项目绑死。所以 Core 里的 `ConfigValidationInterceptor` 用一个 `Dictionary<string, object>` 把可选依赖塞给 Validator，Validator 自己按 key 取。

当前代码里，这个字典是 `ConfigValidationInterceptor.Intercept` 每次拦截业务方法时新建的：

```csharp
var services = new Dictionary<string, object>();
```

然后拦截器为避免 Core 项目直接引用 Config 实现，通过类型名动态解析：

```csharp
Type.GetType("KH.WMS.Config.Abstractions.IConfigResolverContract, KH.WMS.Config")
```

解析成功后放入：

```csharp
services["ConfigService"] = configService;
```

最后调用每个 Validator 时传进去：

```csharp
var errorMessage = validator
    .ValidateAsync(invocation.Arguments, services)
    .GetAwaiter()
    .GetResult();
```

所以 `services` 的值不是开发者在业务 Service 里手动组装的，正常路径是：

1. 业务方法通过接口代理被调用。
2. 方法上有 `[ConfigValidation]`。
3. `ConfigValidationInterceptor` 进入拦截逻辑。
4. 拦截器从当前 Autofac `ILifetimeScope` 解析所有 `IValidator`。
5. 拦截器创建 `Dictionary<string, object>`。
6. 拦截器尝试解析配置服务。
7. 解析成功时写入 `services["ConfigService"]`。
8. 调用 `validator.ValidateAsync(invocation.Arguments, services)`。

目前这个字典里只有一个约定 key：

| Key | 实际类型 | 什么时候有值 | 用途 |
| --- | --- | --- | --- |
| `"ConfigService"` | `IConfigResolverContract` | Config 程序集可加载且实现已注册时 | 读取全局配置、作用域配置、布尔开关等运行时配置 |

Validator 中可以这样取：

```csharp
var configService = services?.TryGetValue("ConfigService", out var svc) == true
    ? svc as IConfigResolverContract
    : null;
```

如果 `services == null`，通常说明这个 Validator 不是通过 `ConfigValidationInterceptor` 调度的，而是被单元测试、临时手动调用或其他代码直接调用了。因为参数默认值是 `null`，手动调用 `validator.ValidateAsync(args)` 可以编译通过，但这时拿不到拦截器注入的上下文服务。

如果 `services` 不为 null 但没有 `"ConfigService"`，通常是以下情况：

- `KH.WMS.Config` 程序集没有被加载。
- `IConfigResolverContract` 的实现没有注册成功。
- `Type.GetType("KH.WMS.Config.Abstractions.IConfigResolverContract, KH.WMS.Config")` 没解析到类型。
- 当前运行环境只加载了 Core/业务模块，没加载配置模块。

### 什么时候要使用 `services`

只有当校验规则依赖“运行时配置”时，才需要用 `services`。

需要使用的典型场景：

- 是否允许混料：读取 `CONTAINER.ALLOW_MIXED_MATERIAL`。
- 是否允许混批：读取 `CONTAINER.ALLOW_MIXED_BATCH`。
- 是否启用批次管理：读取 `CONTAINER.BATCH_MANAGEMENT`。
- 是否要求效期：读取对应配置开关。
- 同一条校验规则在不同仓库、客户、租户或业务范围下有不同开关。

现有 `BatchNoRequiredValidator` 就是这种模式：

```csharp
public async Task<string?> ValidateAsync(
    object?[] args,
    Dictionary<string, object>? services = null)
{
    var configService = GetConfigService(services);
    if (configService == null) return null;

    if (!await configService.ResolveConfigBoolAsync("CONTAINER", "BATCH_MANAGEMENT"))
        return null;

    var binds = args.OfType<List<ContainerBindDto>>().FirstOrDefault();
    if (binds == null) return null;

    foreach (var bind in binds)
    {
        if (string.IsNullOrWhiteSpace(bind.BatchNo))
            return WMSErrorMessages.GetMessage(BATCH_NO_REQUIRED);
    }

    return null;
}

private static IConfigResolverContract? GetConfigService(
    Dictionary<string, object>? services)
{
    if (services == null) return null;
    return services.TryGetValue("ConfigService", out var svc)
        ? svc as IConfigResolverContract
        : null;
}
```

不需要使用的典型场景：

- 只校验参数是否为空。
- 只校验数量是否超出订单行可用量。
- 只查业务表做固定规则判断。
- 规则不受配置开关影响。

例如 `BindDataNotEmptyValidator` 只需要从 `args` 里拿 `List<ContainerBindDto>`，完全不需要 `services`：

```csharp
public Task<string?> ValidateAsync(
    object?[] args,
    Dictionary<string, object>? services = null)
{
    var binds = args.OfType<List<ContainerBindDto>>().FirstOrDefault();
    if (binds == null || binds.Count == 0)
        return Task.FromResult<string?>(WMSErrorMessages.GetMessage(CONTAINER_BIND_DATA_EMPTY));

    return Task.FromResult<string?>(null);
}
```

### `services` 和构造函数注入怎么选

Validator 仍然可以正常使用构造函数注入。比如 `MixedMaterialValidator` 和 `BindQuantityValidator` 都通过构造函数注入了 `ISqlSugarClient`：

```csharp
public class BindQuantityValidator : IValidator
{
    private readonly ISqlSugarClient _db;

    public BindQuantityValidator(ISqlSugarClient db)
    {
        _db = db;
    }
}
```

选择规则：

| 依赖类型 | 推荐方式 | 原因 |
| --- | --- | --- |
| 仓储、数据库、日志、业务 Contract | 构造函数注入 | 正常 DI 依赖，类型明确，测试也清楚 |
| 拦截器统一提供的可选上下文能力 | `services` 字典 | Core 不直接引用上层模块，按 key 解耦 |
| 当前业务方法入参 | `args` | 来自被拦截方法的实参 |

不要把 `services` 当成万能 ServiceProvider 使用。当前它不是 DI 容器，只是拦截器传给 Validator 的少量上下文服务集合。需要什么服务就优先走构造函数注入；只有拦截器统一塞进来的上下文对象，才从 `services` 取。

## 校验链短路细节

一个方法上有多个校验：

```csharp
[ConfigValidation(ValidatorCodes.BIND_DATA_NOT_EMPTY)]
[ConfigValidation(ValidatorCodes.BIND_QUANTITY)]
[ConfigValidation(ValidatorCodes.MIXED_MATERIAL)]
public async Task<ServiceResult> ContainerBindAsync(List<ContainerBindDto> binds)
```

执行时：

1. `BIND_DATA_NOT_EMPTY` 先执行。
2. 如果返回错误消息，立即返回 `ServiceResult.Fail`，后两个校验不执行，原方法不执行。
3. 如果返回 null，继续 `BIND_QUANTITY`。
4. 所有校验返回 null 后，才执行 `ContainerBindAsync` 方法体。

因此顺序应该从“最便宜、最基础”的校验到“更重、需要查库”的校验。空数据校验应放在查库校验前。

## Validator 参数读取模式

Validator 拿到的是 `object?[] args`，不是强类型方法参数。常见写法：

```csharp
var binds = args.OfType<List<ContainerBindDto>>().FirstOrDefault();
if (binds == null) return null;
```

这有两个含义：

- 方法签名变了，Validator 可能取不到参数并直接放行。
- 多个相同类型参数时，`FirstOrDefault` 可能拿错。

如果校验器强依赖参数结构，改业务方法签名时要同步改 Validator。

## 配置缺失时是放行还是失败

现有入库校验器里，如果拿不到 `ConfigService`，通常返回 null 放行。这是为了避免配置服务不可用时校验器直接阻断所有业务。

但新增校验器时要明确策略：

- 配置缺失时按默认允许：适合兼容历史数据。
- 配置缺失时按默认禁止：适合高风险规则。
- 配置缺失时报错：适合必须配置才能执行的流程。

不要无意识地 `return null`，否则规则可能静默失效。

## 什么时候不要用 ConfigValidation

不要把所有校验都塞进 Validator。以下情况更适合写在业务 Service：

- 校验依赖当前事务中刚写入但未提交的数据。
- 校验失败后需要回滚已经执行的多步操作。
- 校验需要返回结构化数据，不只是错误消息。
- 校验和业务状态变更强绑定，例如“提交时自动补齐任务”。

`ConfigValidation` 的定位是“方法执行前的可插拔前置校验链”。

## 排查校验没执行

按下面顺序查：

1. 方法是否通过代理调用。
2. 方法上是否加 `[ConfigValidation]`，且加在实现类方法上。
3. 方法返回类型是否是 `Task<ServiceResult>` 或 `Task<ServiceResult<T>>`。
4. Validator 是否注册成 `IValidator`。
5. Validator `Code` 是否和特性字符串一致。
6. `ConfigValidationInterceptor` 是否解析到了该 Validator。
7. Validator 是否因为参数没取到或配置缺失而返回 null。

## 最小示例

### 新增编码常量

```csharp
public static class ValidatorCodes
{
    public const string DEMO_RULE = "DEMO_RULE";
}
```

### 实现 Validator

```csharp
[RegisteredService(WithoutInterceptor = true, ServiceType = typeof(IValidator))]
public class DemoRuleValidator : IValidator
{
    public string Code => ValidatorCodes.DEMO_RULE;

    public async Task<string?> ValidateAsync(object?[] args, Dictionary<string, object>? services = null)
    {
        var request = args.OfType<DemoRequest>().FirstOrDefault();
        if (request == null)
            return "请求不能为空";

        return null;
    }
}
```

### 在业务方法上启用

```csharp
[ConfigValidation(ValidatorCodes.DEMO_RULE)]
public async Task<ServiceResult> SubmitAsync(DemoRequest request)
{
    ...
}
```

## 常见误区与排查清单

- 方法返回 `ApiResponse`：当前配置校验不会短路。
- Validator 没注册成 `IValidator`：拦截器找不到。
- `Code` 和 `[ConfigValidation]` 字符串不一致：校验器不会执行。
- Validator 没加 `WithoutInterceptor=true`：可能产生递归或额外代理问题。
- 校验器找不到配置服务：确认 Config 程序集被加载，`IConfigResolverContract` 实现已注册。
- 多个校验顺序不对：调整 `[ConfigValidation]` 的声明顺序。
- 找不到校验器时当前逻辑会跳过：新增校验后要确认日志和断点，避免静默漏校验。
