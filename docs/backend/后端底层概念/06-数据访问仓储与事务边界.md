# 06 数据访问、仓储与事务边界

## 这个概念解决什么问题

数据访问底座解决的是“业务代码怎么读写数据库，事务在哪里开始和结束”的问题。

KH.WMS 使用 SqlSugar 作为 ORM，在其上封装了：

- `IRepository<TEntity,TKey>`：实体级仓储。
- `IUnitOfWork`：请求作用域内的数据库上下文和事务边界。
- `CrudService<TEntity>`：标准 CRUD 的事务、审计字段、主从表保存。
- `[Transaction]` + `TransactionActionFilter`：Controller/Action 级自动事务。
- `[ConfigDb]`：配置实体自动路由到 SQLite 配置库。

业务开发要重点理解：不是所有写操作都要自己开事务，事务边界应该由“一个业务流程的原子性”决定。

## 什么时候需要看

- 新增 Service，需要访问仓储。
- 一个接口里有多个写操作，需要一起成功或一起失败。
- `CrudService` 已经有事务，但你又手动开了事务。
- 配置表读写到了业务库，或业务表误路由到配置库。
- 事务没回滚、嵌套事务行为不符合预期。
- 需要查询主从表、按明细过滤、直接使用 SqlSugar Queryable。

## 业务开发应该怎么用

### 普通 CRUD 不需要手动开事务

继承 `CrudService<TEntity>` 后，新增、更新、删除、批量删除内部已经处理事务：

- `CreateAsync`：开始事务、钩子、填充审计字段、插入主表、保存从表、提交。
- `UpdateAsync`：开始事务、加载旧数据、钩子、复制字段、保存主从表、提交。
- `DeleteAsync`：开始事务、删除前钩子、级联删除、删除后钩子、提交。

你只需要重写钩子，不要在钩子里再随意 `BeginTransactionAsync`。

### 跨多个写操作的业务流程手动控制事务

例如组盘、请求上架、取消组盘这种流程，通常包含多个仓储和 Contract 调用，需要在 Service 方法里手动：

```csharp
try
{
    await _unitOfWork.BeginTransactionAsync();

    ...

    await _unitOfWork.CommitAsync();
    return ServiceResult.Ok();
}
catch (Exception ex)
{
    await _unitOfWork.RollbackAsync();
    return ServiceResult.Fail(ex.Message);
}
```

如果希望异常交给全局异常 Filter 处理，则 catch 后应 rollback 再 throw，不能吞掉异常。

### Controller/Action 级事务用 `[Transaction]`

当一个 Action 本身就是事务边界，并且业务逻辑不方便放进已有 Service 事务时，可以使用：

```csharp
[Transaction]
[HttpPost("submit")]
public async Task<ApiResponse> Submit(...)
{
    ...
}
```

它不是 Autofac AOP，而是 MVC Filter。只有经过 MVC Action 执行才会生效。

### 读写配置表不要手选连接

配置实体加了 `[ConfigDb]` 后，`RepositoryBase` 会自动把它路由到配置库：

```csharp
_db = typeof(T).GetCustomAttribute<ConfigDbAttribute>() != null
    ? ((SqlSugarClient)db).GetConnection(SqlSugarSetup.ConfigDb)
    : db;
```

业务代码通常只注入 `IRepository<CfgGlobalConfig,long>`，不要自己到处写 `GetConnection(SqlSugarSetup.ConfigDb)`。

## 底层逻辑和实现

### 主库和配置库

`SqlSugarSetup` 注册了两个连接：

- `MainDb`：业务库，数据库类型由配置决定，例如 SQL Server、MySQL。
- `ConfigDb`：配置库，SQLite 文件，默认 `Data/kh-wms-config.db`。

`[ConfigDb]` 标记在配置实体类上，仓储构造时根据实体类型自动路由。

### 仓储提供什么能力

`RepositoryBase<T,TKey>` 提供：

- 单条查询：`GetByIdAsync`、`GetFirstOrDefaultAsync`。
- 列表查询：`GetAllAsync`、`GetListAsync`。
- 分页：`GetPagedListAsync`。
- 新增、更新、删除、批量新增、批量删除。
- 导航属性查询和保存：`GetByIdWithNavAsync`、`AddWithNavAsync`、`UpdateWithNavAsync`、`DeleteWithNavAsync`。
- 按明细过滤主表：`GetListByDetailAsync`、`GetFirstOrDefaultByDetailAsync`。
- 原生 Queryable：`AsQueryable()`。

复杂查询可以用 `AsQueryable()`，但不要绕过仓储到处散落连接选择逻辑。

### `IUnitOfWork` 做什么

`IUnitOfWork` 封装：

- 当前 `IDbContext`。
- 当前事务状态。
- 事务深度。
- 获取仓储。
- Begin、Commit、Rollback。

它是 `Scoped`，同一个请求作用域内共享同一个上下文和事务栈。

### 嵌套事务栈

`SqlSugarDbContext` 用 `Stack<TransactionScope>` 支持嵌套事务：

- 最外层 `BeginTransactionAsync` 才真正调用数据库 `BeginTranAsync`。
- 内层 begin 只增加事务深度。
- 内层 commit 只标记完成并出栈。
- 内层 rollback 会把回滚信号传播到外层。
- 最外层 commit 时如果发现需要回滚，会走 rollback。
- dispose 时如果还有未提交事务，会尝试自动回滚。

所以业务上不要依赖“内层事务独立提交”。这里的嵌套事务是同一个数据库事务里的逻辑嵌套。

### `[Transaction]` 的执行方式

`TransactionAttribute` 实现 `IFilterFactory`，MVC 会创建 `TransactionActionFilter`：

1. 如果 `IUnitOfWork.HasActiveTransaction` 已经为 true，直接执行 Action，不重复管理。
2. 否则开始事务。
3. Action 正常执行且没有异常，提交。
4. Action 有异常，回滚。

这解释了为什么 Service 手动事务和 Action 事务可以共存：Filter 会检测已有事务。

### Action 有 `[Transaction]`，Service 又手动开事务会怎样

这是最容易误判的场景。实际行为取决于事务栈，而不是代码里出现了几次 `BeginTransactionAsync`。

假设 Controller Action 标了 `[Transaction]`，Filter 会先执行：

```text
Action Filter Begin -> 栈深度 1，数据库真正 BeginTran
Action 调用 Service
Service Begin -> 栈深度 2，不再开启新的数据库事务
```

这时 Service 里面的手动事务不是独立数据库事务，只是同一个数据库事务里的内层逻辑范围。

常见分支如下：

| 场景 | Service 内层行为 | Action Filter 最后看到什么 | 最终数据库结果 |
| --- | --- | --- | --- |
| Service 成功并 `CommitAsync` | 内层出栈，栈深度从 2 回到 1，不提交数据库 | Action 无异常 | 外层 `CommitAsync` 才真正提交 |
| Service catch 后 `RollbackAsync`，再重新抛异常 | 内层 rollback 会把外层 scope 标记为 `RequiresRollback` | `executedContext.Exception != null`，Filter 再 rollback 外层 | 数据库回滚 |
| Service catch 后 `RollbackAsync`，再返回失败结果 | 内层 rollback 会把外层 scope 标记为 `RequiresRollback` | Action 没异常，Filter 会尝试 commit | 外层 commit 检测到 `RequiresRollback`，改为 rollback |
| Service 开了内层事务但异常一路抛出，自己没有 rollback | 内层 scope 仍在栈顶 | Filter 只调用一次 rollback，可能只是弹出内层并把外层标记为 `RequiresRollback` | 真实数据库回滚可能延后到 `SqlSugarDbContext.Dispose()`，属于错误写法 |
| Service catch 后不 rollback，直接返回失败结果 | 内层如果已开事务但没配对 commit/rollback，会残留事务栈；如果根本没开内层事务，则没有回滚信号 | Action 没异常 | 可能提交失败分支里的写入，或留下未提交事务等到 dispose 回滚，属于错误写法 |
| Service 内层 `CommitAsync` 后 Action 后续又抛异常 | 内层 commit 只出栈，不提交数据库 | Action 有异常 | 外层 rollback，前面写入一起回滚 |

所以要记住两句话：

- `[Transaction]` 包住 Action 时，Service 手动 `CommitAsync` 不代表数据库已经提交。
- Service 手动 `RollbackAsync` 会通过 `RequiresRollback` 传播给外层，即使最后 Action 返回的是失败对象而不是抛异常，外层也不应该再提交。
- 只要 Service 手动 `BeginTransactionAsync`，就必须在 Service 自己的 `try/catch` 里配对 `CommitAsync` 或 `RollbackAsync`。不要把内层事务栈留给 Action Filter 猜。

业务上更推荐一个流程只选择一个清晰事务边界。薄 Action 组合多个服务时可以用 `[Transaction]`；复杂流程放在 Service 内时更推荐 Service 自己管理事务，并让 Controller 保持薄。

## 三种事务边界对比

| 用法 | 触发位置 | 适合场景 | 成功条件 | 回滚条件 |
| --- | --- | --- | --- | --- |
| `CrudService` 内置事务 | Service 方法内部 | 标准新增、更新、删除、批删 | 方法执行完并 `CommitAsync` | catch 到异常 |
| `[Transaction]` | MVC Action Filter | 整个 Action 是一个事务 | Action 没有异常 | `executedContext.Exception != null` |
| 手动 `IUnitOfWork` | Service 自己控制 | 跨多个仓储/Contract 的复杂流程 | 显式 `CommitAsync` | 显式 `RollbackAsync` 或异常 catch |

选择原则：

- 标准 CRUD：用 `CrudService`，不要重复包。
- 多步骤业务流程：Service 手动事务更清晰。
- 很薄的 Action 组合多个服务：可以考虑 `[Transaction]`。注意“返回失败对象”本身不会让 Filter 回滚；只有异常或 Service 内部已经显式 rollback 并传播了 `RequiresRollback`，最终才会回滚。

## 嵌套事务栈举例

假设外层业务方法和内层 CRUD 都调用了 `BeginTransactionAsync`：

```text
外层 Begin -> 栈深度 1，数据库真正 BeginTran
内层 Begin -> 栈深度 2，不再开启新数据库事务
内层 Commit -> 栈深度回到 1，不提交数据库事务
外层 Commit -> 栈深度 1，数据库真正 CommitTran
```

如果内层 rollback：

```text
外层 Begin -> 栈深度 1
内层 Begin -> 栈深度 2
内层 Rollback -> 标记 RequiresRollback，并传播给外层
外层 Commit -> 发现 RequiresRollback，改为 Rollback
```

这说明内层失败会影响整个外层事务，不存在“内层回滚但外层仍提交”的独立事务语义。

## 配置库路由细节

`RepositoryBase<T,TKey>` 构造时只看实体类型是否有 `[ConfigDb]`：

```csharp
typeof(T).GetCustomAttribute<ConfigDbAttribute>() != null
```

这带来两个实践要求：

- 配置实体必须在类上加 `[ConfigDb]`，否则会进入主业务库。
- 业务实体不要误加 `[ConfigDb]`，否则会进入 SQLite 配置库。

它不是根据 Repository 名称、Service 名称、模块名称判断数据库。

## 仓储和 SqlSugar 直接访问怎么选

优先级建议：

1. 简单实体 CRUD：用 `IRepository<TEntity,long>`。
2. 复杂条件但仍是单实体：用 `_repository.AsQueryable()`。
3. 多实体联查、聚合统计：可以用 SqlSugar Queryable，但放在 Service 或 QueryService。
4. 跨模块可复用查询：做成 Contract 或 QueryService。
5. 不建议在 Controller 里直接用 `ISqlSugarClient`。

如果使用 `ISqlSugarClient` 直接访问配置库，要非常明确原因。普通配置实体读写应交给 `[ConfigDb]` + Repository。

## 并发与事务边界

事务只保证一个数据库事务内的原子性，不自动解决所有并发问题。

例如库存锁定中使用了条件更新：

```csharp
Where(d => d.Id == inventoryDetailId && (d.Qty - d.LockedQty) >= lockQty)
```

这类并发控制是业务 SQL 条件和数据库更新共同保证的，不是仅靠 `IUnitOfWork`。如果你的流程涉及库存、容器、任务等并发资源，要检查是否需要：

- 行锁。
- 条件更新。
- 唯一约束。
- 幂等判断。
- 事务内重新查询。

## 最小示例

### 查询

```csharp
var repo = _unitOfWork.GetRepository<InvInventoryHeader, long>();
var header = await repo.GetFirstOrDefaultAsync(x => x.ContainerCode == containerCode);
```

### 复杂查询

```csharp
var rows = await _repository.AsQueryable()
    .Where(x => x.WarehouseId == warehouseId)
    .OrderBy(x => x.CreatedTime, OrderByType.Desc)
    .ToListAsync();
```

### 手动事务

```csharp
await _unitOfWork.BeginTransactionAsync();
try
{
    await headerRepo.AddAsync(header);
    await detailRepo.AddAsync(details);
    await _unitOfWork.CommitAsync();
}
catch
{
    await _unitOfWork.RollbackAsync();
    throw;
}
```

## 常见误区与排查清单

### 事务没有回滚

- 是否 catch 后返回成功，导致外层认为无异常。
- 是否写操作发生在事务外的另一个连接或上下文。
- 是否用了 `[Transaction]`，但代码不是 MVC Action 调用。
- 是否在 `CrudService` 钩子里另开事务并吞异常。

### 配置表进错库

- 配置实体是否标了 `[ConfigDb]`。
- 仓储是否用的是 `IRepository<CfgXxx,long>`。
- 是否业务代码手动绕过了仓储和路由。

### 嵌套事务误解

- 内层 commit 不等于数据库已经提交。
- 内层 rollback 会让外层最终回滚。
- 不要把嵌套事务当成数据库 savepoint 使用。

### 仓储查询不够用

- 优先使用 `AsQueryable()` 组合 SqlSugar 查询。
- 如果查询是跨模块公共能力，考虑做 QueryService 或 Contract。
- 不要把复杂 SQL 写进 Controller。
