# 13 鉴权、缓存、用户上下文与运行期开关

## 这个概念解决什么问题

这一组能力解决的是运行期横向问题：

- 请求是否允许访问。
- 当前用户是谁。
- Token 是否有效，是否需要刷新。
- 常用数据是否可以缓存。
- License 是否有效。
- 是否开启限流。
- 如何查看接口和 SQL 性能。

这些能力大多是底座自动生效，业务开发需要知道边界和排查入口，但不应该随意改启动管道。

## 什么时候需要看

- 接口返回 401、402 或被鉴权拦截。
- 新接口需要匿名访问。
- 业务 Service 需要当前用户 ID 或用户名。
- `[Cache]` 没生效，或缓存脏数据。
- License 拦截了请求。
- 配了限流但没有生效。
- 想用 MiniProfiler 看慢接口和 SQL。

## 业务开发应该怎么用

### 匿名接口用 `[AllowAnonymous]`

`ApiAuthorizeFilter` 会检查 EndpointMetadata 里是否有 `IAllowAnonymous`。登录、公钥、License 恢复等接口可以匿名。

```csharp
[AllowAnonymous]
[HttpPost("login")]
public async Task<ApiResponse> Login(LoginDto dto)
{
    ...
}
```

普通业务接口不要随意加匿名。

### 获取当前用户用 `IUserContext`

```csharp
public class DemoService(IUserContext userContext)
{
    public Task<ApiResponse> SubmitAsync()
    {
        var userId = userContext.UserId;
        var userName = userContext.UserName;
        ...
    }
}
```

`IUserContext` 从当前请求的 Bearer Token 解析用户信息，同一请求内会缓存解析结果。

后台任务或无 HTTP 上下文场景下，用户信息可能为空，业务代码要能处理。

### 普通缓存用 `ICacheService`

```csharp
var key = $"demo:{id}";
var data = await _cache.GetOrCreateAsync(key, () => LoadDataAsync(id), TimeSpan.FromMinutes(10));
```

适合业务主动控制缓存 key、过期时间、清理时机的场景。

### 方法返回缓存用 `[Cache]`

```csharp
[Cache(Duration = 600, KeyPrefix = "dict:items")]
public async Task<ApiResponse> GetDictItemsAsync(string dictType)
{
    ...
}
```

适合稳定查询，不适合写操作。写操作后要考虑清理相关缓存。

## 底层逻辑和实现

### 鉴权流程

`ApiAuthorizeFilter` 是 MVC 授权 Filter，它的关键逻辑：

1. 如果接口带 `[AllowAnonymous]`，放行。
2. 检查 `HttpContext.User.Identity.IsAuthenticated`。
3. 从 Authorization header 取 Bearer token。
4. 用 `IJwtTokenService` 解析 userId。
5. 从缓存读取当前用户 token。
6. 如果不允许多端登录且缓存 token 与当前 token 不一致，返回账号被踢下线。
7. 如果 token 快过期，生成新 token 和 refresh token，通过响应头返回。

无 token、签名无效、过期等情况会返回 401。

### 多端登录开关

是否允许多端登录来自系统参数缓存：

```text
SYS_ALLOW_MULTI_LOGIN
```

`ApiAuthorizeFilter` 读取的缓存 key 是 `CacheConstants.SysParam.GetKey(SysParamConstants.ALLOW_MULTI_LOGIN)`，实际系统参数编码是 `SYS_ALLOW_MULTI_LOGIN`。

如果值为空，默认允许多端登录。值为 `1` 或 `true` 时允许多端登录；其他禁用值会触发单端登录判断。禁用多端登录时，缓存中的 token 与当前请求 token 不一致，就会返回 401，并提示账号被挤下线。

这个判断只发生在 token 已经通过 JWT Bearer 验签之后，所以它不是“token 无效”的校验，而是“当前用户最新有效 token 是否还是这个”的校验。

### Token 临期刷新

`ApiAuthorizeFilter` 调用 `ShouldRefreshToken(token)` 判断是否需要刷新。需要刷新时：

- 生成新的 access token。
- 生成新的 refresh token。
- 更新缓存里的当前 token。
- 通过响应头返回：
  - `X-Access-Token`
  - `X-Refresh-Token`

代码里对应的常量名是 `HeaderConstants.Authentication.X_ACCESS_TOKEN` 和 `X_REFRESH_TOKEN`，但真正写到 HTTP 响应里的 header 名是 `X-Access-Token`、`X-Refresh-Token`。前端 axios 里读取时通常会被规范成小写，比如 `x-access-token`、`x-refresh-token`。

刷新后，后端会用 `SYS_TOKEN_EXPIRE_MIN` 决定新 access token 在缓存里的过期时间；如果配置缺失或非法，默认使用 30 分钟。业务接口不需要手动刷新 token，只要前端统一拦截响应头并替换本地 token。

### 用户上下文

`UserContext` 提供：

- `UserName`
- `UserId`
- `RoleId`
- `Token`
- `Permissions`
- `MenuType`
- `IsAuthenticated`
- `IsSuperAdmin`
- `HttpContext`

当前实现主要从 Bearer Token 解析用户 ID、用户名和角色。它注册为 `WithoutInterceptor=true`，避免底层上下文服务被 AOP 代理。

需要注意这些字段的真实来源：

| 字段 | 来源 | 使用提醒 |
| --- | --- | --- |
| `Token` | 当前请求 `Authorization: Bearer ...` | 同一请求内只解析一次；无 HTTP 上下文或无 Bearer 头时为 null |
| `UserId` | `IJwtTokenService.GetUserIdFromToken(Token)` | 业务写创建人、修改人时优先用它；后台任务要判空 |
| `UserName` | `IJwtTokenService.GetUsernameFromToken(Token)` | 只表示 token 中的用户名，不会实时查用户表 |
| `RoleId` | token 的角色 claim 里的第一个值 | 多角色场景不能只靠这个做完整权限判断 |
| `IsSuperAdmin` | `RoleId == RoleConstants.SUPER_ADMIN_ROLE_ID` | 只适合判断超级管理员快捷分支 |
| `MenuType` | 请求头是否包含 `uniapp` | 有 `uniapp` header 时返回 1，否则返回 0 |
| `Permissions` | 当前实现初始化为空列表 | 不要假设这里已经自动加载了按钮或菜单权限 |

也就是说，`IUserContext` 是“请求上下文读取器”，不是用户资料服务。需要用户昵称、部门、角色集合、权限集合时，应调用对应业务服务或 Contract，不要把这些逻辑塞进 `UserContext`。

### 缓存服务

`CacheSetup` 注册：

- `ICacheService`
- `IMemoryCacheService`

当前默认实现是内存缓存 `MemoryCacheService`。这意味着缓存是进程内缓存，多个服务实例之间不共享。

`[Cache]` 不是“所有返回值都缓存”。`CachingInterceptor` 的实际限制是：

- 会先找实现方法上的 `[Cache]`，再找接口方法，最后找类上的 `[Cache]`。
- `Enable=false` 时直接跳过。
- 同步方法和 `Task<T>` 都支持命中返回。
- 只有返回值是 `ApiResponse` 且 `ApiResponse.Data != null` 时才会写入缓存。
- 返回 `Task` 但没有泛型结果的写操作，不适合用 `[Cache]`。
- `ApiResponse` 失败但带了 `Data` 时，按当前代码也可能被缓存；错误响应不要随便带可缓存数据。

缓存 key 的生成规则是：

```text
KeyPrefix 或 TargetType.MethodName + ":" + 参数 ToString() 拼接
```

所以复杂 DTO、数组、集合参数如果没有稳定的 `ToString()`，可能出现 key 不可读、key 冲突或“看起来参数变了但 key 没变”的问题。复杂查询更建议手动使用 `ICacheService` 设计 key。

### License 运行期开关

`LicenseValidationMiddleware` 位于自定义中间件链中，异常处理之后、认证授权之前。

它会：

- 读取 `License:Enabled`，默认启用。
- 白名单路径直接放行，例如登录、license 导入、Swagger、健康检查。
- 校验 License。
- 无效时返回 HTTP 402，并返回统一 `ApiResponse(Code=LICENSE_REQUIRED)`。

402 和 RBAC 的 403 是不同语义：402 表示授权许可问题。

当前白名单使用精确或带边界匹配，主要包括：

```text
/api/user/login
/api/user/public-key
/api/user/logout
/api/license/machine-code
/api/license/import
/api/license/info
/api/license/upload
/swagger
/health
/healthchecks
```

`License:Enabled=false` 可以关闭拦截，适合本地开发或测试环境临时排查；生产环境不建议关闭。应用启动时还会通过 License 中间件扩展确保默认 License 状态初始化，所以排查 402 时要看配置、License 文件、机器码和当前请求路径是否命中白名单。

### 限流当前状态

`AddInfrastructure` 中调用了 `AddRateLimiting(configuration)`，注册了限流配置。

但 `MiddlewareSetup` 中的实际启用代码当前是注释状态：

```csharp
//app.UseRateLimiting();
```

所以“配置了 RateLimit 但没有限流效果”时，先确认中间件是否启用。

### MiniProfiler

MiniProfiler 在 `AddMonitoringSetup` 中注册，在 `UseCustomMiddleware` 的 `UseRouting` 之后启用：

```csharp
app.UseMiniProfilerCustom();
```

它会记录：

- 请求性能。
- SQL timing。
- `LoggingInterceptor` 写入的方法步骤。

默认路由是 `/profiler`，是否生产启用由配置决定，默认生产不开启。

## 401 链路拆解

一个业务接口返回 401，可能发生在这些点：

1. 接口没有 `[AllowAnonymous]`。
2. JWT Bearer 中间件没有把 `HttpContext.User` 设置为认证状态。
3. `ApiAuthorizeFilter` 判断 `IsAuthenticated == false`。
4. Header 没有 Bearer token，或 token 格式不对。
5. `IJwtTokenService.GetUserIdFromToken(token)` 返回 null。
6. 多端登录关闭时，缓存 token 和当前 token 不一致。

排查时先区分：

- token 本身无效：通常所有接口都 401。
- 被挤下线：缓存中的用户 token 和当前请求 token 不一致。
- 接口忘记匿名：登录、公钥等接口才需要 `[AllowAnonymous]`。

## 402 链路拆解

License 中间件在认证授权之前执行。返回 402 时：

1. `License:Enabled` 没有关闭。
2. 请求路径不在白名单。
3. `ILicenseService.ValidateLicense()` 返回 null。
4. 中间件写出 `ApiResponse.Fail(ResponseCode.LICENSE_REQUIRED, message)`。
5. HTTP 状态码设置为 402。

所以 402 与用户权限无关，换 token 通常没用。要看 License 文件、机器码、有效期和白名单路径。

## 缓存 key 设计

手动使用 `ICacheService` 时，key 要能表达数据边界：

```text
模块:业务:参数:用户或仓库
```

示例：

```csharp
var key = $"inventory:overview:warehouse:{warehouseId}";
```

如果缓存结果和用户、仓库、角色有关，就必须把这些信息放进 key，或者不要缓存。

`[Cache]` 特性有 `IncludeUser` 属性，但当前 `CachingInterceptor.GenerateCacheKey` 只使用 `KeyPrefix` 和参数拼 key，没有使用 `IncludeUser`。所以涉及用户隔离的数据，不要只依赖 `[Cache(IncludeUser=true)]`，要确认拦截器实现是否支持。

## MiniProfiler 排查路径

慢接口排查可以按这个顺序：

1. 看接口总耗时。
2. 看 SQL timing，确认是不是数据库慢。
3. 看 `LoggingInterceptor` 写入的方法步骤，确认慢在哪个 Service。
4. 如果没有方法步骤，看目标 Service 是否有 `[LogInterceptor]` 或是否经过 AOP。
5. 如果 SQL 很多，看是否存在 N+1 查询。
6. 如果接口总耗时高但 SQL 不高，看外部调用、序列化、文件处理或锁等待。

MiniProfiler 是定位工具，不是业务逻辑依赖。不要在业务代码里根据 MiniProfiler 是否存在决定流程。

## 运行期开关认知边界

| 能力 | 注册了吗 | 当前是否自动生效 | 业务开发该怎么介入 |
| --- | --- | --- | --- |
| JWT 认证 | 是 | 是 | 正常带 Bearer token |
| `ApiAuthorizeFilter` | 是 | 是 | 匿名接口加 `[AllowAnonymous]` |
| `IUserContext` | 是 | 请求内可用 | 注入读取用户信息 |
| 内存缓存 | 是 | 手动或特性触发 | 设计 key、清理缓存 |
| License | 是 | 默认启用 | 排查 402、维护 License |
| 限流 | 配置已注册 | 中间件当前注释 | 需要启用 `UseRateLimiting` |
| MiniProfiler | 是 | 按配置启用 | 排查性能 |

## 常见误区与排查清单

### 接口返回 401

- 是否带 Bearer token。
- token 是否过期或签名无效。
- 接口是否应该匿名，如果是，是否加 `[AllowAnonymous]`。
- 多端登录关闭时，是否被新登录 token 挤下线。

### 接口返回 402

- License 是否启用。
- 请求路径是否在白名单。
- License 文件是否有效。
- 前端是否按 402 跳转到授权恢复页面。

### 当前用户为空

- 是否在 HTTP 请求内。
- Authorization header 是否存在。
- token 是否能解析 userId 和 username。
- 后台任务中不要假设一定有当前用户。

### 缓存没生效或数据脏

- `[Cache]` 是否加在通过代理调用的方法上。
- 返回值是否是 `ApiResponse` 且 `Data != null`。
- key 是否因为参数 `ToString()` 不稳定而冲突。
- 数据更新后是否清理缓存。

### 限流没生效

- `AddRateLimiting` 只是注册配置。
- 当前 `app.UseRateLimiting()` 是注释状态，未真正挂到管道。

### MiniProfiler 看不到数据

- 是否被配置关闭。
- 是否访问了正确的 profiler 路由。
- 生产环境默认不开启。
- 方法级步骤依赖日志拦截器写入。
