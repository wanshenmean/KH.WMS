# 14 策略算法底座

## 这个概念解决什么问题

策略算法底座解决的是“业务流程需要按配置选择算法，而不是硬编码某个策略类”的问题。

在 WMS 里，很多决策不是固定逻辑：

- 入库上架应该去哪个库区、巷道、接驳口。
- 上架时如何分配具体货位。
- 出库时如何选择库存批次。
- 下架、波次等流程如何按仓库、库区、单据类型差异化。

KH.WMS 用 `IPolicyEngine` 统一执行策略。业务模块只提供上下文并调用策略引擎，不直接 new 或注入具体策略实现。

## 什么时候需要看

- 入库、出库、任务流程需要调用策略。
- 要新增一种策略类型或策略实现。
- 策略链配置后没有生效。
- 想知道为什么某个策略没有被执行。
- 想避免在业务代码里硬编码“优先整托、先进先出”等算法。
- 排查策略返回为空、失败、跳过。

## 业务开发应该怎么用

### 按策略类型执行

最常见入口：

```csharp
var result = await _policyEngine.ExecuteAsync(PolicyType.Putaway, context);
```

适合由数据库策略链配置决定具体执行哪些策略。

### 按策略编码执行

```csharp
var result = await _policyEngine.ExecuteAsync("FIFO_INVENTORY", context);
```

适合明确要执行某个注册策略的场景。业务流程里不建议滥用，避免绕过策略链配置。

### 按策略链编码执行

```csharp
var result = await _policyEngine.ExecuteChainAsync("PUTAWAY_DEFAULT", context);
```

适合已经明确链编码的场景。

### 构造上下文

业务模块负责构造 `PolicyContext`：

```csharp
var context = new PolicyContext
{
    WarehouseId = warehouseId,
    ZoneId = zoneId,
    DocType = docType,
    MaterialId = materialId,
    BusinessCode = docNo
};

context.SetData(StrategyParams.PutawayInput.INBOUND_STATION_CODE, inboundStationCode);
```

上下文越准确，策略链匹配和策略执行越准确。

### 处理结果

```csharp
var result = await _policyEngine.ExecuteAsync(PolicyType.Putaway, context);
if (!result.IsSuccess)
    return ServiceResult.Fail(result.ErrorMessage);

var putawayResult = result.Output as PutawayResult;
if (putawayResult == null)
    return ServiceResult.Fail("上架策略返回结果为空");
```

不要假设策略一定成功，也不要直接强转后使用。

## 底层逻辑和实现

### 策略类型

`PolicyType` 定义当前策略分类：

- `Putaway`：上架策略。
- `LocationAllocation`：货位分配策略。
- `Picking`：下架策略。
- `InventoryAllocation`：库存分配策略。
- `Wave`：波次策略。
- `OutboundAllocation`：历史兼容枚举，标记为废弃，新出库流程不应使用。

业务调用时优先使用这些枚举，而不是直接依赖具体策略类名。

### 策略如何注册

`Program.cs` 注册了：

```csharp
builder.RegisterModule(new StrategyAutofacModule());
```

`StrategyAutofacModule` 会：

1. 扫描算法程序集里所有非抽象 `IPolicyStrategy` 实现。
2. 把每个策略注册为 Singleton。
3. 容器构建完成后解析所有策略。
4. 调用 `PolicyRegistry.RegisterStrategy(strategy)`。

所以新增策略实现后，要确认它在算法程序集里、实现了 `IPolicyStrategy`、能被扫描到。

### 策略引擎执行顺序

`PolicyEngine.ExecuteAsync(PolicyType, context)`：

1. 记录策略类型和上下文。
2. 尝试从数据库配置构建策略链。
3. 如果找到可用策略链，按策略链执行。
4. 如果找不到数据库策略链，回退为使用注册中心里该类型的所有策略。

### 数据库策略链匹配

构建数据库策略链时，会查询启用的策略链配置，并按上下文匹配。

匹配维度包括：

- 仓库。
- 库区。
- 单据类型。
- 默认链。
- 优先级。

大致原则是：匹配越精确、优先级越高，越先被选中。

### 策略链步骤

匹配到策略链后：

1. 查询策略链步骤。
2. 只取启用的步骤。
3. 按 `StepNo` 排序。
4. 通过 `StrategyConfigId` 找策略配置。
5. 用配置里的 `RuleCode` 到 `PolicyRegistry` 找策略实例。
6. 将步骤参数和策略默认参数传给策略链。

如果步骤配置了但策略实现没有注册，会被跳过并记录日志。

### Pipeline 模式

当前 `LocationAllocation` 会启用 `PipelineMode`。这表示每个策略会基于上游结果继续筛选，适合货位候选集逐步过滤。

业务调用方只需要知道：同样是策略链，不同 `PolicyType` 的执行模式可能不同，具体由策略引擎内部决定。

## 策略链匹配细节

`PolicyEngine.SelectBestChain` 会给匹配维度打分。当前关键维度包括：

- 仓库匹配。
- 库区匹配。
- 单据类型匹配。
- 优先级。
- 默认链。

如果某个链配置了仓库 ID，但请求上下文没有 `WarehouseId`，这条链会被视为不匹配。配置了库区或单据类型也同理。

所以调用策略前，业务方要尽量把上下文补完整：

```csharp
var context = new PolicyContext
{
    WarehouseId = warehouseId,
    ZoneId = zoneId,
    DocType = docType,
    MaterialId = materialId,
    BusinessCode = docNo
};
```

不要只传 `PolicyType`，却不给仓库、单据类型等关键上下文，然后期待系统自动命中精确链。

## 策略配置与策略实现的关系

数据库配置里通常存的是策略链和步骤，而真正执行的是代码里的 `IPolicyStrategy` 实例。

执行映射链路：

```text
CfgStrategyChainConfig
  -> CfgStrategyChainStep.StrategyConfigId
  -> CfgStrategyConfig.RuleCode
  -> PolicyRegistry.GetStrategy(ruleCode)
  -> IPolicyStrategy.ExecuteAsync
```

如果配置里有步骤但策略没执行，重点查：

- `StrategyConfigId` 是否存在。
- `RuleCode` 是否和策略实现的编码一致。
- 策略实现是否被 `StrategyAutofacModule` 注册进 `PolicyRegistry`。
- 步骤是否启用。

## 策略调用方的责任

业务模块调用策略前后要承担这些责任：

- 准备足够上下文。
- 处理失败结果。
- 校验 `Output` 类型。
- 把策略结果转换成业务请求，例如任务创建请求。
- 在业务事务中保存后续数据。

策略引擎不负责：

- 自动创建任务。
- 自动更新单据状态。
- 自动扣库存。
- 自动提交业务事务。

策略只做决策，业务流程负责执行决策。

## 策略失败处理建议

不要把策略失败吞掉后继续流程：

```csharp
var result = await _policyEngine.ExecuteAsync(PolicyType.Putaway, context);
if (!result.IsSuccess)
    return ServiceResult.Fail($"上架策略执行失败: {result.ErrorMessage}");
```

如果策略返回成功但 `Output` 类型不对，也应视为业务失败：

```csharp
if (result.Output is not PutawayResult output)
    return ServiceResult.Fail("上架策略返回结果类型错误");
```

继续执行会导致后续任务缺少目标库区、接驳口或货位，产生更难排查的数据问题。

## 新增策略实现检查表

新增一个策略实现后，至少确认：

1. 类在 `KH.WMS.Algorithms` 程序集内。
2. 实现 `IPolicyStrategy`。
3. `PolicyType` 正确。
4. `RuleCode` 或策略编码和配置表一致。
5. `IsEnabled` 为 true。
6. `IsApplicableAsync` 不会误判跳过。
7. 策略无状态，适合 Singleton 注册。
8. 需要的参数能从 `PolicyContext` 或步骤参数里取到。
9. 数据库策略链步骤已启用且排序正确。

## 最小示例

入库组盘请求上架时调用上架策略：

```csharp
var context = new PolicyContext
{
    WarehouseId = input.WarehouseId,
    BusinessCode = input.BusinessCode,
    DocType = input.DocType,
    MaterialId = input.MaterialId,
};

if (!string.IsNullOrWhiteSpace(input.InboundStationCode))
{
    context.SetData(
        StrategyParams.PutawayInput.INBOUND_STATION_CODE,
        input.InboundStationCode);
}

var putawayResult = await _policyEngine.ExecuteAsync(PolicyType.Putaway, context);
if (!putawayResult.IsSuccess)
{
    return ServiceResult.Fail(putawayResult.ErrorMessage);
}
```

## 常见误区与排查清单

### 策略没有执行

- 策略实现是否实现 `IPolicyStrategy`。
- 策略是否在算法程序集里，能被 `StrategyAutofacModule` 扫描。
- `PolicyType` 是否匹配。
- 策略的 `IsEnabled` 是否为 true。
- `IsApplicableAsync` 是否返回 true。

### 策略链配置没生效

- 数据库是否有对应 `chainType` 的启用策略链。
- 仓库、库区、单据类型是否和 `PolicyContext` 一致。
- 是否配置了默认链。
- 步骤是否启用。
- `StrategyConfigId` 是否存在。
- `RuleCode` 是否能在 `PolicyRegistry` 找到注册策略。

### 返回结果为空

- 策略执行是否成功。
- `Output` 类型是否符合调用方预期。
- 策略链中是否没有任何可用策略。
- 是否回退到默认注册策略后得到不同结果。

### 不要这样做

- 不要在业务 Service 里直接 new 某个策略类。
- 不要把策略优先级写死在业务 if/else 里。
- 不要继续使用已废弃的 `OutboundAllocation` 做新出库流程。
- 不要把策略配置失败吞掉后继续执行关键流程。
